                                                                                                                                                          Abstract

                                                                                                                                                          摘要

                                                                                                                                                          Pre-trained and frozen LLMs can effectively map simple scene re-arrangement instructions to programs over a robot's visuomotor functions through appropriate few-shot example prompting. To parse open-domain natural language and adapt to a user's idiosyncratic procedures, not known during prompt engineering time, fixed prompts fall short. In this paper, we introduce HELPER, an embodied agent equipped with as external memory of language-program pairs that parses free-form human-robot dialogue into action programs through retrieval-augmented LLM prompting: relevant memories are retrieved based on the current dialogue, instruction, correction or VLM description, and used as in-context prompt examples for LLM querying. The memory is expanded during deployment to include pairs of user's language and action plans, to assist future inferences and personalize them to the user's language and routines. HELPER sets a new state-of-the-art in the TEACh benchmark in both Execution from Dialog History (EDH) and Trajectory from Dialogue (TfD), with 1.7x improvement over the previous SOTA for TfD. Our models, code and video results can be found in our project's website: helper-agent-llm.github.io. 预训练且冻结的 LLM 通过恰当的少样本示例提示，能够有效地将简单的场景重排指令映射为机器人视觉运动功能上的程序。要解析开放领域的自然语言，并适配用户在提示工程阶段未知的个人化流程，固定提示就力有未逮。本文提出 HELPER，这是一种具备外部语言-程序对记忆的具身代理：它通过检索增强的 LLM 提示，将自由形式的人机对话解析为动作程序。系统会基于当前对话、指令、纠正或 VLM 描述检索相关记忆，并将其作为上下文提示示例供 LLM 查询。部署过程中，记忆会持续扩展，以纳入用户语言与行动计划的配对，从而帮助未来推断，并将其个性化到用户的语言与日常习惯。HEPER 在 TEACh 基准的对话执行（EDH）和对话轨迹生成（TfD）两项上均达到新的最先进水平：相较此前 TfD 的 SOTA，提升 1.7 倍。我们的模型、代码和视频结果可在项目网站查看：helper-agent-llm.github.io。

                                                                                                                                                          1 Introduction

                                                                                                                                                          1 引言

                                                                                                                                                          Parsing free-form human instructions and human-robot dialogue into task plans that a robot can execute is challenging due to the open-endedness of environments and procedures to accomplish, and to the diversity and complexity of language humans use to communicate their desires. Human language often contains long-term references, questions, errors, omissions, or references and descriptions of routines specific to a particular user (Tellex et al., 2011; Liang, 2016; Klein and Manning, 2003). Instructions need to be interpreted in the environmental context in which they are issued, and plans need to adapt in a closed-loop to execution failures. 将自由形式的人类指令和人机对话解析为机器人可执行的任务计划，具有挑战性，因为环境和完成这些任务的流程具有开放性，而人类用于表达意图的语言也多样而复杂。自然语言中常包含长期指代、问题、错误、遗漏，或针对特定用户的惯例性指代和描述（Tellex et al., 2011; Liang, 2016; Klein and Manning, 2003）。指令需要结合其给出的环境上下文来理解，计划也需要在执行失败后以闭环方式自适应调整。

                                                                                                                                                          Large Language Models (LLMs) trained on Internet-scale text can parse language instructions to task plans with appropriate plan-like or code-like prompts, without any finetuning of the language model, as shown in recent works (Ahn et al., 2022; Liang et al., 2022; Zeng et al., 2022; Huang et al., 2022b; Singh et al., 2022a). The state of the environment is provided as a list of objects and their spatial coordinates, or as a free-form text description from a vision-language model (Liang et al., 2022; Liu et al., 2023b; Wu et al., 2023a; Ahn et al., 2022). Using LLMs for task planning requires engineering a prompt that includes a description of the task for the LLM to perform, a robot API with function documentation and expressive function names, environment and task instruction inputs, and a set of in-context examples for inputs and outputs for the task (Liang et al., 2022). These methods are not trained in the domain of interest; rather they are prompt-engineered having in mind the domain at hand. 近期工作表明，大语言模型（LLMs）在互联网规模文本上训练后，无需对语言模型进行任何微调，只需使用合适的类计划或类代码提示，就能将语言指令解析为任务计划（Ahn et al., 2022; Liang et al., 2022; Zeng et al., 2022; Huang et al., 2022b; Singh et al., 2022a）。环境状态以物体及其空间坐标的列表形式提供，或由视觉语言模型给出的自由形式文本描述提供（Liang et al., 2022; Liu et al., 2023b; Wu et al., 2023a; Ahn et al., 2022）。将LLMs用于任务规划，需要设计一个提示，其中包括供LLM执行的任务描述、带函数文档和具表现力的函数名的机器人API、环境与任务指令输入，以及用于任务输入和输出的一组上下文示例（Liang et al., 2022）。这些方法并非在目标领域上训练；相反，它们是在提示工程中针对当前领域量身设计的。

                                                                                                                                                          How can we extend LLM-prompting for semantic parsing and task planning to open-domain, freeform instructions, corrections, human-robot dialogue, and users' idiosyncratic routines, not known at prompt engineering time? The prompts used for the domain of tabletop rearrangement are already approaching the maximum context window of widely used LLMs (Singh et al., 2022a; Liang et al., 2022). Even as context window size grows, more prompt examples result in larger attention operations and cause an increase in both inference time and resource usage. 我们如何将用于语义解析和任务规划的LLM提示扩展到开放领域、自由形式指令、更正、人机对话，以及在提示工程时并未知晓的用户个性化惯例？用于桌面重排领域的提示已接近广泛使用的LLMs的最大上下文窗口（Singh et al., 2022a; Liang et al., 2022）。即使上下文窗口继续增大，更多的提示示例也会带来更大的注意力计算，并导致推理时间和资源消耗增加。

                                                                                                                                                          To this end, we introduce HELPER (Human-instructable Embodied Language Parsing via Evolving Routines), a model that uses retrieval-augmented situated prompting of LLMs to parse free-form dialogue, instructions and corrections from humans and vision-language models to programs over a set of parameterized visuomotor routines. HELPER is equipped with an external nonparametric key-value memory of language - program pairs. HELPER uses its memory to retrieve relevant in-context language and action program examples, and generates prompts tailored to the current language input. HELPER expands its memory with successful executions of user specific procedures; it then recalls them and adapts them in future interactions with the user. HELPER uses pre-trained vision-language models (VLMs) to diagnose plan failures in language format, and uses these to retrieve similar failure cases with solutions from its memory to seed the prompt. To execute a program predicted by the LLM, HELPER combines successful practices of previous home embodied agents, such as semantic and occupancy map building (Chaplot et al., 2020a; Blukis et al., 2022; Min et al., 2021), LLM-based common sense object search (Inoue and Ohashi, 2022), object detection and tracking with off-the-shelf detectors (Chap-lot et al., 2020b), object attribute detection with VLMs (Zhang et al., 2022), and verification of action preconditions during execution. 为此，我们提出HELPER（Human-instructable Embodied Language Parsing via Evolving Routines），一种利用检索增强的情境提示来解析人类和视觉语言模型给出的自由形式对话、指令与更正，并将其转换为一组参数化视听运动例程上的程序的模型。HELPER配备了一个外部的非参数化语言-程序对键值记忆。HELPER利用其记忆检索相关的上下文内语言和动作程序示例，并生成针对当前语言输入的提示。HELPER通过成功执行用户特定流程来扩展其记忆；随后在未来与该用户的交互中回忆并调整这些流程。HELPER使用预训练的视觉语言模型（VLMs）以语言形式诊断计划失败，并据此检索记忆中带有解决方案的相似失败案例来为提示提供种子。为了执行LLM预测的程序，HELPER结合了以往家庭具身智能体的成功做法，例如语义和占用地图构建（Chaplot et al., 2020a; Blukis et al., 2022; Min et al., 2021）、基于LLM的常识性物体搜索（Inoue and Ohashi, 2022）、使用现成检测器进行物体检测与跟踪（Chap-lot et al., 2020b）、使用VLMs进行物体属性检测（Zhang et al., 2022），以及执行过程中动作前提条件的验证。

                                                                                                                                                          We test HELPER on the TEACh benchmark (Pad-makumar et al., 2021), which evaluates agents in their ability to complete a variety of long-horizon household tasks from RGB input given natural language dialogue between a commander (the instruction-giving user) and a follower (the instruction-seeking user). We achieve a new state-of-the-art in the TEACh Execution from Dialog History and Trajectory-from-Dialogue settings, improving task success by 1.7x and goal-condition success by 2.1x compared to prior work in TfD. By further soliciting and incorporating user feedback, HELPER attains an additional 1.3x boost in task success. Our work is inspired by works in the language domain (Perez et al., 2021; Schick and Schütze, 2020; Gao et al., 2020; Liu et al., 2021) that retrieve in-context prompt examples based on the input language query for NLP tasks. HELPER extends this capability to the domain of instructable embodied agents, and demonstrates the potential of memory-augmented LLMs for semantic parsing of open-ended free-form instructive language into an expandable library of programs. 我们在 TEACh 基准（Pad-makumar 等，2021）上测试 HELPER，该基准评估智能体在仅凭 RGB 输入、并在指挥者（给出指令的用户）与跟随者（寻求指令的用户）之间的自然语言对话条件下，完成各种长时域家务任务的能力。相较于 TfD 领域的既有工作，我们在 TEACh 的“基于对话历史的执行（Execution from Dialog History）”与“基于对话轨迹的生成（Trajectory-from-Dialogue）”设置中取得了新的最先进结果：任务成功提升 1.7x，目标条件达成成功提升 2.1 倍。通过进一步征询并整合用户反馈，HELPER 在任务成功上又获得了额外 1.3 倍的增益。我们的工作受到语言领域研究的启发（Perez 等，2021；Schick 和 Schütze，2020；Gao 等，2020；Liu 等，2021）：这些研究会根据输入语言查询，为 NLP 任务检索上下文提示示例。HELPER 将这一能力扩展到可指令的具身智能体领域，并展示了记忆增强的大语言模型用于将开放式、自由形式的指令性语言进行语义解析，从而映射到可扩展的程序库的潜力。

                                                                                                                                                          2 Related Work

                                                                                                                                                          2 相关工作

                                                                                                                                                          Instructable Embodied Agents Significant strides have been made by training large neural networks to jointly map instructions and their sensory contexts to agent actions or macro-actions using imitation learning (Anderson et al., 2018b; Ku et al., 2020; Anderson et al., 2018a; Savva et al., 2019; Gervet et al., 2022; Shridhar et al., 2020; Cao et al.; Suglia et al., 2021; Fan et al., 2018; Yu et al., 2020; Brohan et al., 2022; Stone et al., 2023; Yu et al., 2023). Existing approaches differ—among others-in the way the state of the environment is communicated to the model. Many methods map RGB image tokens and language inputs directly to actions or macro-actions (Pashevich et al., 2021; Wijmans et al., 2020; Suglia et al., 2021; Krantz et al., 2020). Other methods map language instructions and linguistic descriptions of the environment's state in terms of object lists or objects spatial coordinates to macro-actions, foregoing visual feature description of the scene, in a attempt to generalize better (Liang et al., 2022; Singh et al., 2022a; Chaplot et al., 2020a; Min et al., 2021; Liu et al., 2022a; Murray and Cakmak, 2022; Liu et al., 2022b; Inoue and Ohashi, 2022; Song et al., 2022; Zheng et al., 2022; Zhang et al., 2022; Huang et al., 2022b, 2023; Ahn et al., 2022; Zeng et al., 2022; Huang et al., 2022a). Some of these methods fine-tune language models to map language input to macro-actions, while others prompt frozen LLMs to predict action programs, relying on the emergent in-context learning property of LLMs to emulate novel tasks at test time. Some methods use natural language as the output format of the LLM (Wu et al., 2023a; Song et al., 2022; Blukis et al., 2022; Huang et al., 2022b), and others use code format (Singh et al., 2022a; Liang et al., 2022; Huang et al., 2023). HELPER prompts frozen LLMs to predict Python programs over visuo-motor functions for parsing dialogue, instructions and corrective human feedback. 可指令具身代理 通过训练大型神经网络，使其能够借助模仿学习，将指令与感知语境共同映射到智能体动作或宏动作（Anderson et al., 2018b; Ku et al., 2020; Anderson et al., 2018a; Savva et al., 2019; Gervet et al., 2022; Shridhar et al., 2020; Cao et al.; Suglia et al., 2021; Fan et al., 2018; Yu et al., 2020; Brohan et al., 2022; Stone et al., 2023; Yu et al., 2023），取得了显著进展。现有方法的差异在于——例如——环境状态如何被传达给模型。许多方法将 RGB 图像特征标记与语言输入直接映射为动作或宏动作（Pashevich et al., 2021; Wijmans et al., 2020; Suglia et al., 2021; Krantz et al., 2020）。还有一些方法将语言指令以及环境状态的语言描述（用物体列表或物体的空间坐标来表述）映射为宏动作，而不对场景进行视觉特征描述，以便在泛化上做得更好（Liang et al., 2022; Singh et al., 2022a; Chaplot et al., 2020a; Min et al., 2021; Liu et al., 2022a; Murray and Cakmak, 2022; Liu et al., 2022b; Inoue and Ohashi, 2022; Song et al., 2022; Zheng et al., 2022; Zhang et al., 2022; Huang et al., 2022b, 2023; Ahn et al., 2022; Zeng et al., 2022; Huang et al., 2022a）。其中一些方法微调语言模型，将语言输入映射到宏动作；而另一些方法通过提示冻结的 LLM 来预测动作程序，利用 LLM 的涌现式上下文学习能力，在测试时模拟新任务。一些方法使用自然语言作为 LLM 的输出格式（Wu et al., 2023a; Song et al., 2022; Blukis et al., 2022; Huang et al., 2022b），另一些则使用代码格式（Singh et al., 2022a; Liang et al., 2022; Huang et al., 2023）。HELPER 用提示词驱动冻结的 LLM，在视觉-运动功能上预测 Python 程序，用于解析对话、指令以及纠正性的人工反馈。

                                                                                                                                                          The work closest to HELPER is LLM Planner (Song et al., 2022) which uses memory-augmented prompting of pretrained LLMs for instruction following. However, it differs from HELPER in several areas such as plan memory expansion, VLM-guided correction, and usage of LLMs for object search. Furthermore, while Singh et al. (2022b) frequently seeks human feedback, HELPER requests feedback only post full task execution and employs Visual-Language Models (VLMs) for error feedback, reducing user interruptions. 与 HELPER 最接近的工作是 LLM Planner（Song et al., 2022），它通过带记忆增强的提示来引导预训练 LLM 完成指令跟随。然而，HEL PER 在多个方面与之不同，例如计划记忆的扩展、VLM 引导的纠错，以及使用 LLM 进行物体搜索。此外，尽管 Singh et al.（2022b）经常寻求人类反馈，HELPER 只在完整任务执行之后才请求反馈，并使用视觉-语言模型（VLM）进行错误反馈，从而减少用户中断。

                                                                                                                                                          Numerous simulation environments exist for evaluating home assistant frameworks, including Habitat (Savva et al., 2019), GibsonWorld (Shen et al., 2021), ThreeDWorld (Gan et al., 2022), and AI2THOR (Kolve et al., 2017). ALFRED (Shrid-har et al., 2020) and TEACh (Padmakumar et al., 2021) are benchmarks in the AI2THOR environment (Kolve et al., 2017), measuring agents' competence in household tasks through natural language. Our research focuses on the 'Trajectory from Dialogue' (TfD) evaluation in TEACh, mirroring ALFRED but with greater task and input complexity. 用于评估家庭助理框架的模拟环境有很多，包括 Habitat（Savva et al., 2019）、GibsonWorld（Shen et al., 2021）、ThreeDWorld（Gan et al., 2022）和 AI2THOR（Kolve et al., 2017）。ALFRED（Shrid-har et al., 2020）和 TEACh（Padmakumar et al., 2021）是在 AI2THOR 环境中的基准（Kolve et al., 2017），通过自然语言来衡量智能体在家庭任务中的能力。我们的研究聚焦于 TEACh 中的“对话生成轨迹”（Trajectory from Dialogue, TfD）评测：它与 ALFRED 类似，但任务与输入复杂度更高。

                                                                                                                                                          Prompting LLMs for action prediction and visual reasoning Since the introduction of few-shot prompting by (Brown et al., 2020), several approaches have improved the prompting ability of LLMs by automatically learning prompts (Lester et al., 2021), chain of thought prompting (Nye et al.; Gao et al., 2022; Wei et al., 2022; Wang et al., 2022; Chen et al., 2022; Yao et al., 2023) and retrieval-augmented LLM prompting (Nakano et al., 2021; Shi et al., 2023; Jiang et al., 2023) for language modeling, question answering, and long-form, multi-hop text generation. HELPER uses memory-augmented prompting by retrieving and integrating similar task plans into the prompt to facilitate language parsing to programs. 用于动作预测与视觉推理的LLM提示自从（Brown et al., 2020）提出少样本提示以来，已有多种方法通过自动学习提示（Lester et al., 2021）、链式思维提示（Nye et al.; Gao et al., 2022; Wei et al., 2022; Wang et al., 2022; Chen et al., 2022; Yao et al., 2023）以及检索增强的LLM提示（Nakano et al., 2021; Shi et al., 2023; Jiang et al., 2023）来提升语言建模、问答和长篇、多跳文本生成的提示能力。HELPER通过检索并将相似的任务计划整合到提示中来进行记忆增强提示，从而促进将语言解析为程序。

                                                                                                                                                          LLMs have been used as policies in Minecraft to predict actions (Wang et al., 2023b,a), error correction (Liu et al., 2023b), and for understanding instruction manuals for game play in some Atari games (Wu et al., 2023b). They have also significantly improved text-based agents in text-based simulated worlds (Yao et al., 2022; Shinn et al., 2023; Wu et al., 2023c; Richards, 2023). ViperGPT (Surís et al., 2023), and CodeVQA (Sub-ramanian et al., 2023) use LLM prompting to decompose referential expressions and questions to programs over simpler visual routines. Our work uses LLMs for planning from free-form dialogue and user corrective feedback for home task completion, a domain not addressed in previous works. LLM已在Minecraft中被用作策略，用于预测动作（Wang et al., 2023b,a）、进行错误纠正（Liu et al., 2023b），并在部分Atari游戏中通过理解游戏操作的说明手册来辅助游戏（Wu et al., 2023b）。它们也显著提升了基于文本的模拟世界中的文本代理（Yao et al., 2022; Shinn et al., 2023; Wu et al., 2023c; Richards, 2023）。ViperGPT（Surís et al., 2023）和CodeVQA（Sub-ramanian et al., 2023）使用LLM提示，将指称表达和问题分解为更简单视觉例程之上的程序。我们的工作则使用LLM基于自由形式对话和用户纠错反馈进行规划，以完成家庭任务——这是先前研究未涉及的领域。

                                                                                                                                                          3 Method

                                                                                                                                                          3 方法

                                                                                                                                                          HELPER is an embodied agent designed to map human-robot dialogue, corrections and VLM descriptions to actions programs over a fixed API of parameterized navigation and manipulation primitives. Its architecture is outlined in Figure 2. At its heart, it generates plans and plan adjustments by querying LLMs using retrieval of relevant language-program pairs to include as in-context examples in the LLM prompt. The generated programs are then sent to the EXECUTOR module, which translates each program step into specific navigation and manipulation action. Before executing each step in the program, the EXECUTOR verifies if the necessary preconditions for an action, such as the robot already holding an object, are met. If not, the plan is adjusted according to the current environmental and agent state. Should a step involve an undetected object, the EXECUTOR calls on the LOCATOR module to efficiently search for the required object by utilizing previous user instructions and LLMs' common sense knowledge. If any action fails during execution, a VLM predicts the reason for this failure from pixel input and feeds this into the PLANNER for generating plan adjustments. HELPER 是一种具身智能体，旨在将人机对话、纠正信息以及 VLM 描述映射为基于固定 API 的动作程序，API 由参数化的导航与操作原语构成。其架构如图 2 所示。核心在于：通过查询 LLM，并检索相关的语言-程序对，将其作为情境示例纳入 LLM 提示，从而生成计划及对计划的调整。随后，生成的程序会被发送至执行器（EXECUTOR）模块，该模块将程序的每一步转换为具体的导航与操作动作。在执行程序中的每一步之前，EXECUTOR 会检查执行某项动作所需的前提条件是否满足，例如机器人是否已持有某个物体。若不满足，则会根据当前环境与智能体状态对计划进行调整。如果某一步涉及未被检测到的物体，EXECUTOR 会调用定位器（LOCATOR）模块，以利用先前的用户指令及 LLM 的常识知识高效搜索所需物体。若执行过程中任何动作失败，VLM 将根据像素输入预测失败原因，并将其反馈给规划器（PLANNER），以生成计划调整。

                                                                                                                                                          3.1 PLANNER：检索增强的LLM规划

                                                                                                                                                          Given an input I consisting of a dialogue segment, instruction, or correction, HELPER uses memory-augmented prompting of frozen LLMs to map the input into an executable Python program over a parametrized set of manipulation and navigation primitives G∈{G manipulation ∪G navigation } that the EXECUTOR can perform (e.g., goto(X), pickup(X), slice(X), ...). Our action API can be found in Section D of the Appendix. 给定一个由对话片段、指令或修正组成的输入I，HELPER通过对冻结LLM进行记忆增强提示，将输入映射为一个可执行的Python程序，该程序基于一组参数化的操作与导航原语G∈{G manipulation ∪G navigation }，由EXECUTOR执行（例如，goto(X)、pickup(X)、slice(X)等）。我们的动作API见附录D节。

                                                                                                                                                          HELPER maintains a key-value memory of language - program pairs, as shown in Figure 3A. Each language key is mapped to a 1D vector using an LLM's frozen language encoder. Given current context I ,the model retrieves the top- K keys,i.e., the keys with the smallest L2 distance with the embedding of the input context I ,and adds the corresponding language - program pairs to the LLM prompt as in-context examples for parsing the current input I . The LLM is tasked to generate a Python program over parameterized primitives G .                                                                                                                                                           3.1.1 Memory Expansion

                                                                                                                                                          3.1.1 记忆扩展

                                                                                                                                                          The key-value memory of HELPER can be continually expanded with successful executions of instructions to adapt to a user's specific routines, as shown in Figure 1. An additional key-value pair is added with the language instruction paired with the execution plan if the user indicates the task was successful. Then, HELPER can recall this plan and adapt it in subsequent interactions with the user. For example, if a user instructs HELPER one day to "Perform the Mary cleaning. This involves cleaning two plates and two cups in the sink", the user need only say "Do the Mary cleaning" in future interactions, and HELPER will retrieve the previous plan, include it in the examples section of the prompt, and query the LLM to adapt it accordingly. The personalization capabilities of HELPER are evaluated in Section 4.4. HELPER 的键值记忆可以通过成功执行指令而不断扩展，以适应用户的具体习惯，如图 1 所示。若用户表明该任务成功，则会把“语言指令-执行计划”这一组键值对新增到记忆中。随后，HELPER 能够回忆该计划，并在与用户后续互动中进行相应调整。例如，如果用户有一天指示 HELPER“进行玛丽清洁。这包括清洗水槽里的两个盘子和两个杯子”，那么在未来的互动中，用户只需说“做玛丽清洁”，HELPER 就会检索先前的计划，把它纳入提示词的示例部分，并向 LLM 询问据此进行调整。HELPER 的个性化能力在第 4.4 节进行评估。

                                                                                                                                                          3.1.2 Incorporating user feedback

                                                                                                                                                          3.1.2 融入用户反馈

                                                                                                                                                          A user's feedback can improve a robot's performance, but requesting feedback frequently can deteriorate the overall user experience. Thus, we enable HELPER to elicit user feedback only when it has completed execution of the program. Specifically, it asks "Is the task completed to your satisfaction? Did I miss anything?" once it believes it has completed the task. The user responds either that the task has been completed (at which point HELPER stops acting) or points out problems and corrections in free-form natural language, such as, "You failed to cook a slice of potato. The potato slice needs to be cooked.". HELPER uses the language feedback to re-plan using the PLANNER. We evaluate HELPER in its ability to seek and utilize user feedback in Section 4.3. 用户的反馈可以提升机器人的性能，但频繁索取反馈会削弱整体的用户体验。因此，我们让 HELPER 仅在其完成程序执行后才征求用户反馈。具体来说，当它认为任务已完成时，会问：“任务是否已按你的满意度完成？我还有遗漏的吗？”。用户要么回复任务已完成（此时 HELPER 停止行动），要么以自由形式的自然语言指出问题并提出纠正，例如：“你没有把一片土豆做熟。土豆片需要被煮熟。” HELPER 会利用这种语言反馈，通过 PLANNER 重新规划。我们在第 4.3 节评估 HELPER 获取并利用用户反馈的能力。

                                                                                                                                                          3.1.3 Visually-Grounded Plan Correction using Vision-Language Models

                                                                                                                                                          3.1.3 使用视觉-语言模型的视觉引导式计划修正

                                                                                                                                                          Generated programs may fail for various reasons, such as when a step in the plan is missed or an object-of-interest is occluded. When the program fails, HELPER uses a vision-language model (VLM) pre-trained on web-scale data, specifically the ALIGN model (Jia et al., 2021), to match the current visual observation with a pre-defined list of textual failure cases, such as an object is blocking you from interacting with the selected object, as illustrated in Figure 4. The best match is taken to be the failure feedback F . The PLANNER module then retrieves the top- K most relevant error correction examples, each containing input dialogue, failure feedback, and the corresponding corrective program, from memory based on encodings of input I and failure feedback F from the VLM. The LLM is prompted with the the failed program step, the predicted failure description F from the VLM, the in-context examples, and the original dialogue segment I . The LLM outputs a self-reflection as to why the failure occurred, and generates a program over manipulation and navigation primitives G ,and an additional set of corrective primitives G corrective  (e.g.,step-back(),move-to-an-alternate-viewpoint()), ...). This program is sent to the ExECUTOR for execution. 生成的程序可能因多种原因失败，例如计划中的某一步被遗漏，或关注的目标被遮挡。当程序失败时，HELPER 使用一个在大规模网页数据上预训练的视觉-语言模型（VLM），具体为 ALIGN 模型（Jia et al., 2021），将当前的视觉观测与一份预定义的文本故障案例列表进行匹配，例如：某个物体阻挡了你与所选目标进行交互，如图 4 所示。最佳匹配对应的故障反馈为 F 。随后，PLANNER 模块基于来自 VLM 的对输入 I 与故障反馈 F 的编码，从记忆中检索前 K 个最相关的错误修正示例；每个示例都包含输入对话、故障反馈以及相应的纠正程序。LLM 的提示中包括失败程序的步骤、VLM 预测的故障描述 F、上下文示例以及原始对话片段 I 。LLM 会输出关于失败为何发生的自我反思，并生成一个针对操作与导航原语的程序 G ，以及一组额外的纠正原语 G corrective （例如，step-back(),move-to-an-alternate-viewpoint()), ...）。该程序随后被发送给 EXECUTOR 进行执行。

                                                                                                                                                          3.2 Executor: Scene Perception, Pre-Condition Checks, Object Search and Action Execution

                                                                                                                                                          3.2 执行器：场景感知、前置条件检查、物体搜索与动作执行

                                                                                                                                                          The Executor module executes the predicted Python programs in the environment, converting the code into low-level manipulation and navigation actions, as shown in Figure 2. At each time step, the EXECUTOR receives an RGB image and obtains an estimated depth map via monocular depth estimation (Bhat et al., 2023) and object masks via an off-the-shelf object detector (Dong et al., 2021). 执行器模块在环境中执行预测的 Python 程序，将代码转换为低层级的操作与导航动作，如图 2 所示。每个时间步，执行器接收一幅 RGB 图像，并通过单目深度估计（Bhat et al., 2023）获取估计深度图，通过现成的目标检测器（Dong et al., 2021）获取物体掩码。

                                                                                                                                                          3.2.1 Scene and object state perception

                                                                                                                                                          3.2.1 场景与目标状态感知

                                                                                                                                                          Using the depth maps, object masks, and approximate egomotion of the agent at each time step, the EXECUTOR maintains a 3D occupancy map and object memory of the home environment to navigate around obstacles and keep track of previously seen objects, similar to previous works (Sarch et al., 2022). Objects are detected in every frame and are merged into object instances based on closeness of the predicted 3D centroids. Each object instance is initialized with a set of object state attributes (cooked, sliced, dirty, ...) by matching the object crop against each attribute with the pre-trained ALIGN model (Jia et al., 2021). Object attribute states are updated when an object is acted upon via a manipulation action. 通过使用深度图、目标掩膜以及每个时间步智能体的近似自运动估计，执行器（EXECUTOR）维护家庭环境的三维占据地图和目标记忆，以便绕开障碍并追踪先前看到的物体，方式类似于以往工作（Sarch 等，2022）。每一帧都会检测目标，并根据预测的三维质心之间的接近程度将其合并为具体的目标实例。每个目标实例都会借助预训练的 ALIGN 模型（Jia 等，2021）将目标裁剪区域与各个属性进行匹配，从而初始化一组目标状态属性（已烹饪、已切片、脏污……）。当物体在操控动作作用下发生交互时，会更新其属性状态。

                                                                                                                                                          3.2.2 Manipulation and navigation pre-condition checks

                                                                                                                                                          3.2.2 操作与导航前置条件检查

                                                                                                                                                          The EXECUTOR module verifies the pre-conditions of an action before the action is taken to ensure the action is likely to succeed. In our case, these constraints are predefined for each action (for example, the agent must first be holding a knife to slice an object). If any pre-conditions are not satisfied, the EXECUTOR adjusts the plan accordingly. In more open-ended action interfaces, an LLM's common sense knowledge can be used to infer the preconditions for an action, rather than pre-defining them. 执行器（EXECUTOR）会在执行某个动作之前先验证该动作的前置条件，以确保动作更可能成功。在我们的设置中，这些约束为每个动作预先定义（例如，代理在切割物体前必须先拿着刀）。如果任何前置条件未满足，EXECUTOR 会相应调整计划。在更开放的动作接口中，也可以利用 LLM 的常识知识来推断动作的前置条件，而无需事先预定义。

                                                                                                                                                          3.2.3 Locator: LLM-based common sense object search

                                                                                                                                                          3.2.3 定位器：基于LLM的常识性物体搜索

                                                                                                                                                          When HELPER needs to find an object that has not been detected before, it calls on the LOCATOR module. The Locator prompts an LLM to suggest potential object search location for the EXECUTOR to search nearby, e.g. "search near the sink" or "search in the cupboard". The LOCATOR prompt takes in the language I (which may reference the object location, e.g., "take the mug from the cupboard" ) and queries the LLM to generate proposed locations by essentially parsing the instruction as well as using its common sense. Based on these predictions, HELPER will go to the suggested locations if they exist in the semantic map (e.g., to the sink) and search for the object-of-interest. The LOCATOR's prompt can be found in Section D of the Appendix. 当 HELPER 需要寻找以前未被检测到的物体时，它会调用定位器（LOCATOR）模块。定位器会提示一个 LLM，给出可能的物体搜索位置，以便执行器（EXECUTOR）在附近进行搜索，例如“在水槽附近寻找”或“在橱柜里寻找”。定位器的提示会输入语言 I（可能引用物体位置，例如“从橱柜里拿起马克杯”），并向 LLM 查询，通过本质上解析指令，同时结合其常识来生成建议位置。基于这些预测，如果建议位置在语义地图中存在（例如水槽），HElPER 就会前往相应位置搜索目标物体。定位器的提示可在附录 D 中找到。

                                                                                                                                                          4.1 Evaluation on the TEACh dataset

                                                                                                                                                          4.1 在 TEACh 数据集上的评估

                                                                                                                                                          Dataset The dataset comprises over 3,000 human-human, interactive dialogues, geared towards completing household tasks within the AI2-THOR simulation environment (Kolve et al., 2017). We evaluate on the Trajectory from Dialogue (TfD) evaluation variant, where the agent is given a dialogue segment at the start of the episode. The model is then tasked to infer the sequence of actions to execute based on the user's intents in the dialogue segment, ranging from MAKE COFFEE to PREPARE BREAKFAST. We show examples of such dialogues in Figures 1 & 3. We also test on the Execution from Dialogue History (EDH) task in TEACh, where the TfD episodes are partitioned into "sessions". The agent is spawned at the start of one of the sessions and must predict the actions to reach the next session given the dialogue and action history of the previous sessions. The dataset is split into training and validation sets. The validation set is divided into 'seen' and 'unseen' rooms based on their presence in the training set. Validation 'seen' has the same room instances but different object locations and initial states than any episodes in the training set. At each time step, the agent obtains an egocentric RGB image and must choose an action from a specified set to transition to the next step, such as pickup(X), turn left(), etc. Please see Appendix Section G for more details on the simulation environment. 数据集 该数据集包含超过 3,000 段人-人交互对话，旨在在 AI2-THOR 模拟环境中完成家居任务（Kolve et al., 2017）。我们采用“来自对话的轨迹”（TfD）评测变体：在每个回合开始时，智能体会被提供一段对话片段。随后，模型需要根据该对话片段中用户的意图推断要执行的动作序列，从制作咖啡（MAKE COFFEE）到准备早餐（PREPARE BREAKFAST）。这类对话示例见图 1 和图 3。我们还在 TEACh 上测试“来自对话历史的执行”（EDH）任务：将 TfD 回合划分为若干“会话”。智能体在某个会话开始时被生成，并需在已知前述会话的对话与动作历史的情况下，预测动作以到达下一个会话。数据集被划分为训练集和验证集。验证集会根据训练集中是否出现过进行“已见（seen）”与“未见（unseen）”房间划分。已见验证的房间实例与训练集相同，但对象位置和初始状态与训练集中的任一回合不同。在每个时间步，智能体会获得一个以自身视角的 RGB 图像，并必须从给定动作集合中选择一种动作，以切换到下一步，例如 pickup(X)、turn left() 等。有关模拟环境的更多细节，请参见附录 G。

                                                                                                                                                          Evaluation metrics Following evaluation practises for the TEACh benchmark, we use the following two metrics: 1. Task success rate (SR), which refers to the fraction of task sessions in which the agent successfully fulfills all goal conditions. 2. Goal condition success rate (GC), which quantifies the proportion of achieved goal conditions across all sessions. Both of these metrics have corresponding path length weighted (PLW) variants. In these versions, the agent incurs penalties for executing a sequence of actions that surpasses the length of the reference path annotated by human experts. 评测指标 按照 TEACh 基准的评测惯例，我们使用以下两个指标：1. 任务成功率（SR），指智能体成功满足所有目标条件的任务会话所占比例。2. 目标条件成功率（GC），用于衡量在全部会话中实现的目标条件的比例。上述两项指标均对应带路径长度加权（PLW）的变体。在这些版本中，智能体若执行的动作序列超过人类专家标注的参考路径长度，将受到惩罚。

In future work we plan to integrate early 2D features into persistent 3D scene feature representations for more accurate 3D object detection.

                                                                                                                                                          2. 3D 感知瓶颈。HELPER 依赖 2D 目标检测器以及用于 3D 目标定位的深度 3D 反投影。我们观察到，在 HELPER 中使用真实分割，TEACh 成功率提升了 2 倍。未来工作中，我们计划将早期的 2D 特征集成到持久的 3D 场景特征表示中，以实现更准确的 3D 目标检测。

                                                                                                                                                          1. Cost from LLM querying. GPT-4 API is the most accurate LLM used in HELPER and incurs a significant cost. NLP research in model compression may help decreasing these costs, or finetuning smaller models with enough input-output pairs.

                                                                                                                                                          2. 来自 LLM 查询的成本。GPT-4 API 是 HELPER 中使用的最准确的 LLM，但代价很高。模型压缩方面的 NLP 研究可能有助于降低这些成本，或在足够的输入输出样本下对更小的模型进行微调。

                                                                                                                                                          1. Multimodal (vision and language) memory retrieval. Currently, we use a text bottleneck in our environment state descriptions. Exciting future directions include exploration of visual state incorporation to the language model and partial adaptation of its parameters. A multi-modal approach to the memory and plan generation would help contextualize the planning more with the visual state.

                                                                                                                                                          2. 多模态（视觉与语言）记忆检索。目前，我们在环境状态描述中使用的是文本瓶颈。令人期待的未来方向包括探索将视觉状态融入语言模型，以及对其参数进行部分适配。面向记忆与计划生成的多模态方法将有助于让规划更充分地结合视觉状态。

                                                                                                                                                          Last, to follow human instructions outside of simulation environments our model would need to interface with robot closed-loop policies instead of abstract manipulation primitives, following previous work (Liang et al., 2022). 最后，为了在仿真环境之外执行对人的指令，我们的模型需要与机器人闭环策略进行接口对接，而不是使用抽象的操作原语，遵循既有工作（Liang et al., 2022）。

                                                                                                                                                          6 Conclusion

                                                                                                                                                          6 结论

                                                                                                                                                          We presented HELPER, an instructable embodied agent that uses memory-augmented prompting of pre-trained LLMs to parse dialogue segments, instructions and corrections to programs over action primitives, that it executes in household environments from visual input. HELPER updates its memory with user-instructed action programs after successful execution, allowing personalized interactions by recalling and adapting them. It sets a new state-of-the-art in the TEACh benchmark. Future research directions include extending the model to include a visual modality by encoding visual context during memory retrieval or as direct input to the LLM. We believe our work contributes towards exciting new capabilities for instructable and conversable systems, for assisting users and personalizing human-robot communication. 我们提出 HELPER，这是一种可操作指令的具身代理，它通过对预训练 LLM 进行记忆增强式提示，将对话片段、指令与对程序的修正解析为对动作原语的可执行计划，并从视觉输入在家庭环境中执行。HEPPER 在成功执行后，会使用用户指令的动作程序更新其记忆，从而通过回忆与适配实现个性化交互。它在 TEACh 基准上达到了新的最先进水平。未来研究方向包括：扩展模型以引入视觉模态，例如在记忆检索时对视觉上下文进行编码，或将其作为 LLM 的直接输入。我们相信，我们的工作为具备可操作指令与可对话的系统带来令人振奋的新能力，帮助用户并实现人机沟通的个性化。

                                                                                                                                                          The three instances where the PLANNER made errors in the user personalization experiment (Section 4.4) involved logical mistakes or inappropriate alterations to parts of the original plan that were not requested for modification. For instance, in one case - involving a step to modify the plan from making one coffee to two coffees - the PLANNER includes placing two mugs in the coffee maker simultaneously, which is not a valid plan. In the other two instances, the PLANNER omits an object from the original plan that was not mentioned in the modification. 在用户个性化实验（第4.4节）中，PLANNER 发生错误的三处案例，分别源于逻辑失误或对原始计划部分内容的不恰当改动，而这些改动并未被要求修改。比如，其中一例涉及将计划从做一杯咖啡改为做两杯咖啡：PLANNER 同时在咖啡机中放入两个杯子，这不是一个有效的计划。另两例中，PLANNER 都从原始计划里遗漏了并未在改动说明中提及的对象。

                                                                                                                                                          B User Feedback Details

                                                                                                                                                          B 用户反馈详情

                                                                                                                                                          In the user feedback evaluation, once the agent has indicated completion of the task from the original input dialogue, the agent will query feedback from the user. If the simulator indicates success of the task, the agent will end the episode. If the simulator indicates the task is not successful, feedback will be given to the agent for additional planning. This feedback is programatically generated from the TEACh simulator metadata, which gives us information about if the task is successful, and what object state changes are missing in order to complete the task (e.g., bread slice is not toasted, etc.). For each object state that is incorrect, we form a sentence of the following form: "You failed to complete the subtask: subtask. For the object object: description of desired object state." We combine all subtask sentences to create the feedback. HELPER follows the same pipeline (including examples, retrieval, planning, etc.) to process the feedback as with the input dialogue in the normal TfD evaluation. We show experiments with one and two user feedback requests in Section 4.3 of the main paper (a second request is queried if the first user feedback fails to produce task success). 在用户反馈评估中，一旦智能体从原始对话中表明任务已完成，它将向用户征求反馈。如果模拟器指示任务成功，智能体将结束该回合。如果模拟器指示任务未成功，将把反馈提供给智能体以便进行额外规划。该反馈由 TEACh 模拟器元数据以程序方式生成：它告诉我们任务是否成功，以及为完成任务还缺少哪些物体状态变化（例如，面包片未烤制等）。对于每一个不正确的物体状态，我们都会形成如下句子：“你未能完成子任务：子任务。针对物体：所需物体状态的描述。”我们将所有子任务句子合并以生成反馈。HELPER 也遵循相同的处理流程（包括示例、检索、规划等），以与正常 TfD 评估中对输入对话的处理方式一致以处理反馈。我们在主论文第 4.3 节展示了一次和两次用户反馈请求的实验（若第一次用户反馈未能产生任务成功，则会追问第二次请求）。

                                                                                                                                                          G Simulation environment

                                                                                                                                                          G 仿真环境

                                                                                                                                                          The TEACh dataset builds on the Ai2thor simulation environment (Kolve et al., 2017). At each time step the agent may choose from the following actions: Forward(), Backward(), Turn Left(), Turn Right(), Look Up(), Look Down(), Strafe Left(), Strafe Right(), Pickup(X), Place(X), Open(X), Close(X), ToggleOn(X), ToggleOff(X), Slice(X), and Pour(X), where X refers an object specified via a relative coordinate (x,y) on the egocentric RGB frame. Navigation actions move the agent in discrete steps. We rotate in the yaw direction by 90 degrees, and rotate in the pitch direction by 30 degrees. The RGB and depth sensors are at a resolution of 480x480, a field of view of 90 degrees, and lie at a height of 0.9015 meters. The agent's coordinates are parameterized by a single (x,y,z) coordinate triplet with x and z corresponding to movement in the horizontal plane and y reserved for the vertical direction. The TEACh benchmark allows a maximum of 1000 steps and 30 API failures per episode. TEACh 数据集建立在 Ai2thor 仿真环境（Kolve et al., 2017）之上。在每个时间步，智能体可以从以下动作中选择：Forward()、Backward()、Turn Left()、Turn Right()、Look Up()、Look Down()、Strafe Left()、Strafe Right()、Pickup(X)、Place(X)、Open(X)、Close(X)、ToggleOn(X)、ToggleOff(X)、Slice(X) 和 Pour(X)，其中 X 指通过自我中心 RGB 帧上的相对坐标 (x,y) 指定的对象。导航动作使智能体以离散步长移动。我们在 yaw 方向旋转 90 度，在 pitch 方向旋转 30 度。RGB 和深度传感器的分辨率为 480x480，视场角为 90 度，安装高度为 0.9015 米。智能体的坐标由一个 (x,y,z) 坐标三元组参数化，其中 x 和 z 对应水平平面内的移动，y 则保留给垂直方向。TEACh 基准每个回合最多允许 1000 步和 30 次 API 失败。

                                                                                                                                                          H Executor details

                                                                                                                                                          H 执行器详情

                                                                                                                                                          H.1 Semantic mapping and planning

                                                                                                                                                          H.1 语义映射与规划

                                                                                                                                                          Obstacle map HELPER maintains a 2D overhead occupancy map of its environment ∈RH×W that it updates at each time step from the input RGB-D stream. The map is used for exploration and navigation in the environment. 障碍物地图 HELPER 会维护一个二维俯视占据信息地图 ∈RH×W，用于刻画其环境，并在每个时间步基于输入 RGB-D 流进行更新。该地图用于环境中的探索与导航。

                                                                                                                                                          At every time step t ,we unproject the input depth maps using intrinsic and extrinsic information of the camera to obtain a 3D occupancy map registered to the coordinate frame of the agent, similar to earlier navigation agents (Chaplot et al., 2020a). The 2D overhead maps of obstacles and free space are computed by projecting the 3D occupancy along the height direction at multiple height levels and summing. For each input RGB image, we run a SOLQ object segmentor (Dong et al., 2021) (pretrained on COCO (Lin et al., 2014) then finetuned on TEACh rooms) to localize each of 116 semantic object categories. For failure detection, we use a simple matching approach from Min et al. (2021) to compare RGB pixel values before and after taking an action. 在每个时间步 t，我们使用相机的内参和外参对输入深度图进行反投影，得到与智能体坐标系配准的三维占据信息图，类似于较早的导航智能体（Chaplot et al., 2020a）。将三维占据信息沿高度方向在多个高度层级投影并求和，可计算得到二维俯视障碍物与空闲空间地图。对于每一帧输入 RGB 图像，我们运行一个 SOLQ 目标分割器（Dong et al., 2021）（先在 COCO（Lin et al., 2014）上预训练，再在 TEACh 场景上微调），以定位 116 个语义目标类别中的每一个。为检测失败，我们采用 Min et al.（2021）提出的一种简单匹配方法，对比执行动作前后 RGB 像素值。

                                                                                                                                                          Object location and state tracking We maintain an object memory as a list of object detection 3D centroids and their predicted semantic labels {[(X,Y,Z)i,ℓi∈{1…N}],i=1..K} ,where K is the number of objects detected thus far. The object centroids are expressed with respect to the coordinate system of the agent, and, similar to the semantic maps, updated over time using egomotion. We track previously detected objects by their 3D centroid C∈R3 . We estimate the centroid by taking the 3D point corresponding to the median depth within the segmentation mask and bring it to a common coordinate frame. We do a simple form of non-maximum suppression on the object memory, by comparing the euclidean distance of centroids in the memory to new detected centroids of the same category, and keep the one with the highest score if they fall within a distance threshold. 目标位置与状态跟踪 我们维护一个目标记忆，将目标检测得到的三维质心及其预测语义标签存为列表 {[(X,Y,Z)i,ℓi∈{1…N}],i=1..K}，其中 K 表示到目前为止检测到的目标数量。质心相对于智能体坐标系表示，并且与语义地图类似，会随时间利用自运动信息更新。我们通过三维质心 C∈R3 来跟踪先前检测到的目标。我们通过在分割掩码内取中深度对应的三维点来估计质心，并将其转换到统一的坐标系。我们在目标记忆上做一种简单形式的非极大值抑制：将记忆中质心的欧氏距离与同类别新检测到的质心进行比较，若距离落入阈值内，则保留得分最高的那个。

                                                                                                                                                          For each object in the object memory, we maintain an object state dictionary with a pre-defined list of attributes. These attributes include: category label, centroid location, holding, detection score, can use, sliced, toasted, clean, cooked. For the binary attributes, these are initialized by sending the object crop, defined by the detector mask, to the VLM model, and checking its match to each of [f"The {object_category} is {attribute}", f"The {object_category} is not {attribute}"]. We found that initializing these attributes with the VLM gave only a marginal difference to initializing them to default values in the TEACh benchmark, so we do not use it for the TEACh evaluations. However, we anticipate a general method beyond dataset biases of TEACh would much benefit from such vision-based attribute classification. 对于目标记忆中的每个目标，我们维护一个目标状态字典，并包含一组预定义属性列表。这些属性包括：类别标签、质心位置、是否持有、检测得分、是否可用、是否切开、是否烤制、是否清洁、是否烹饪。对于二值属性，我们通过把由检测器掩码定义的目标裁剪块送入 VLM 模型，并检查它与下列任意项的匹配：[f"The {object_category} is {attribute}", f"The {object_category} is not {attribute}"]。我们发现用 VLM 初始化这些属性，相比在 TEACh 基准中用默认值初始化，只带来很小的差异，因此我们不在 TEACh 评估中使用它。不过，我们预期在超越 TEACh 数据集偏置的一般方法上，基于视觉的属性分类会带来很大收益。

                                                                                                                                                          Exploration and path planning HELPER explores the scene using a classical mapping method. We take the initial position of the agent to be the center coordinate in the map. We rotate the agent in-place and use the observations to instantiate an initial map. Second, the agent incrementally completes the maps by randomly sampling an unexplored, traversible location based on the 2D occupancy map built so far, and then navigates to the sampled location, accumulating the new information into the maps at each time step. The number of observations collected at each point in the 2D occupancy map is thresholded to determine whether a given map location is explored or not. Unexplored positions are sampled until the environment has been fully explored, meaning that the number of unexplored points is fewer than a predefined threshold. 探索与路径规划 HELPER 使用经典的映射方法来探索场景。我们将智能体的初始位置设为地图中的中心坐标。首先将智能体原地旋转，并利用观测来生成一个初始地图。随后，智能体会基于迄今为止构建的二维占据信息地图，随机采样一个尚未探索且可通行的位置，逐步补全地图；然后导航到该采样位置，并在每个时间步将新信息累积到地图中。二维占据信息地图中每个位置的观测次数会与阈值进行比较，以判断该位置是否已被探索。持续采样未探索的位置，直到环境被完全探索完成，也就是未探索点的数量少于预定义阈值。

                                                                                                                                                          To navigate to a goal location, we compute the geodesic distance to the goal from all map locations using graph search (Inoue and Ohashi, 2022) given the top-down occupancy map and the goal location in the map. We then simulate action sequences and greedily take the action sequence which results in the largest reduction in geodesic distance. 为导航到目标位置，我们在给定俯视占据信息地图以及地图中的目标位置的情况下，使用图搜索（Inoue and Ohashi, 2022）从所有地图位置计算到目标的测地距离。然后我们模拟一系列动作，并贪心选择能带来测地距离最大降幅的动作序列。
