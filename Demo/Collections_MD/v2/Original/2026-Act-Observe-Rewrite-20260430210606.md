# Act-Observe-Rewrite: Multimodal Coding Agents as In-Context Policy Learners for Robot Manipulation
# Act-Observe-Rewrite：作为机器人操作上下文策略学习者的多模态编码智能体


Vaishak Kumar
Vaishak Kumar


General Bionix, Inc.
General Bionix, Inc.


Abstract. Can a multimodal language model learn to manipulate physical objects by reasoning about its own failures—without gradient updates, demonstrations, or reward engineering? We argue the answer is yes, under conditions we characterise precisely. We present Act-Observe-Rewrite (AOR), a framework in which an LLM agent improves a robot manipulation policy by synthesising entirely new executable Python controller code between trials, guided by visual observations and structured episode outcomes. Unlike prior work that grounds LLMs in pre-defined skill libraries or uses code generation for one-shot plan synthesis, AOR makes the full low-level motor control implementation the unit of LLM reasoning, enabling the agent to change not just what the robot does, but how it does it. The central claim is that interpretable code as the policy representation creates a qualitatively different kind of in-context learning from opaque neural policies: the agent can diagnose systematic failures and rewrite their causes. We validate this across three robosuite manipulation tasks and report promising results, with the agent achieving high success rates without demonstrations, reward engineering, or gradient updates.
摘要。多模态语言模型能否在无需梯度更新、演示数据或奖励工程的情况下，通过推理自身失败来学习操纵物理对象？我们认为答案是肯定的，且是在我们精确界定的条件下。我们提出了 Act-Observe-Rewrite (AOR) 框架，在该框架中，LLM 智能体通过在试验间合成全新的可执行 Python 控制器代码来改进机器人操作策略，并由视觉观察和结构化的情节结果提供指导。与以往将 LLM 锚定在预定义技能库或使用代码生成进行一次性计划合成的研究不同，AOR 将完整的底层电机控制实现作为 LLM 推理的单元，使智能体不仅能改变机器人的动作，还能改变其动作方式。核心主张是：将可解释的代码作为策略表示，创造了一种与不透明神经策略截然不同的上下文学习方式：智能体能够诊断系统性故障并重写其成因。我们在三个 robosuite 操作任务中验证了该方法，并报告了令人振奋的结果：智能体在无需演示、奖励工程或梯度更新的情况下实现了高成功率。


## 1. Introduction
## 1. 引言


Robot manipulation is entering an era of foundation model integration. Vision-language-action models trained on internet-scale data can generalise across hundreds of tasks zero-shot [3, 4]. Yet two significant challenges persist: (1) when these models fail on a specific deployment configuration, diagnosing why and adapting without retraining remains largely unsolved; and (2) the sheer cost of pretraining data and compute places such models out of reach for researchers who want to iterate quickly on novel manipulation tasks.
机器人操作正进入一个基础模型集成的时代。在互联网规模数据上训练的视觉 - 语言 - 动作模型可以零样本泛化处理数百个任务 [3, 4]。然而，两个重大挑战仍然存在：（1）当这些模型在特定部署配置上失败时，诊断失败原因并在不重新训练的情况下进行调整，在很大程度上仍未得到解决；（2）预训练数据和计算的高昂成本使得想要快速迭代处理新操作任务的研究人员难以使用此类模型。


A complementary tradition has emerged in language agent research. Reflexion [24] showed that an LLM can improve performance on decision-making and programming tasks purely through verbal self-reflection - storing episode summaries in memory and using them as additional context in subsequent trials, with no weight updates. ReAct [32] demonstrated that interleaving reasoning traces with environment actions significantly outperforms imitation or RL baselines in grounded settings. Voyager [29] built an open-ended lifelong agent in Minecraft whose skill library grows entirely from LLM-generated executable code.
在语言智能体研究中出现了一种互补的传统。Reflexion [24] 表明，大语言模型（LLM）可以纯粹通过口头自我反思来提高决策和编程任务的性能——将情节摘要存储在内存中，并在后续试验中用作额外的上下文，而无需更新权重。ReAct [32] 证明，在实际场景中，将推理过程与环境动作交织进行的表现明显优于模仿或强化学习（RL）基线。Voyager [29] 在《我的世界》中构建了一个开放式的终身智能体，其技能库完全由大语言模型生成的可执行代码发展而来。


These results suggest a different paradigm for robot learning: instead of training a policy, write one-and rewrite it when it fails. The appeal is clear: no data collection, no training loops, full interpretability, and direct application of the LLM's world knowledge about geometry, physics, and programming. The question is whether this paradigm is viable for continuous physical manipulation, where success depends on precise kinematics, noisy vision, workspace geometry, and object physics-none of which admit the kind of clean natural-language characterisation that makes verbal reflection effective in text-based environments.
这些结果暗示了机器人学习的一种不同范式：与其训练策略，不如编写策略——并在失败时重写它。其吸引力显而易见：无需数据收集，无需训练循环，具备完全的可解释性，并能直接应用 LLM 关于几何、物理和编程的世界知识。问题在于，这种范式对于连续的物理操作是否可行，因为物理操作的成功依赖于精确的运动学、有噪声的视觉、工作空间几何结构和物体物理特性——这些都不具备使语言反思在文本环境中有效的清晰自然语言特征。


This paper addresses that question directly. We introduce Act-Observe-Rewrite (AOR), a two-timescale framework: the robot executes an episode under a Python controller; between episodes, a multimodal LLM reasons over key-frame images and structured outcome data, diagnoses failure modes, and outputs an entirely new controller class. The new controller is dynamically compiled, validated, and loaded for the next trial.
本文直接探讨了这一问题。我们引入了 Act-Observe-Rewrite (AOR)，这是一个双时间尺度框架：机器人在 Python 控制器下执行一个情节；在情节之间，多模态 LLM 对关键帧图像和结构化结果数据进行推理，诊断故障模式，并输出一个全新的控制器类。新控制器被动态编译、验证并加载以用于下一次试验。


The central claim of this paper is not a performance number. It is an architectural claim: that making the full controller implementation the unit of LLM reasoning—rather than a skill selector, a parameter vector, or a reward function—qualitatively changes what can be learned in-context. Specifically:
本文的核心主张并非性能指标，而是一个架构主张：将完整的控制器实现作为 LLM 推理的单元——而不是技能选择器、参数向量或奖励函数——从本质上改变了上下文学习的内容。具体而言：


- The LLM can diagnose the cause of a failure in terms of code logic ("the y-coordinate is negated in the back-projection, which is wrong for OpenGL convention"), not just observe that failure occurred.
- LLM 可以从代码逻辑层面诊断故障原因（例如“y 坐标在反投影中被取反，这不符合 OpenGL 约定”），而不仅仅是观察到故障发生。


- The LLM can make architectural changes-adding phases, changing the target computation, introducing geometric corrections-that go far beyond parameter tuning.
- LLM 可以进行架构层面的更改——增加阶段、改变目标计算、引入几何校正——这些远超参数调整的范畴。


- The LLM can reason about physical constraints-workspace limits, sensor biases, grip geometry—and encode that reasoning as executable corrections.
- LLM 可以推理物理约束——工作空间限制、传感器偏差、抓取几何结构——并将这种推理编码为可执行的修正。


Contribution: The AOR framework: a general architecture for code-synthesis reflexive learning in physical manipulation. Implemented with Claude Code as the coding agent, AOR achieves 100% on two tasks and 91% on a third. The experiments are reported honestly: the residual failures on Stack are an observed shortcoming—the agent identified gripper contact with the target cube as the cause but failed to discover a placement strategy that avoids it, even though one likely exists.
贡献：AOR 框架：一种用于物理操作中代码合成反射学习的通用架构。通过以 Claude Code 作为编码智能体实现，AOR 在两项任务中达到 100% 成功率，在第三项任务中达到 91%。实验结果如实报告：Stack 任务中的残留故障是一个已观察到的缺陷——智能体识别出抓取器与目标立方体的接触是原因，但未能发现一种避免该接触的放置策略，尽管这种策略很可能存在。


## 2. Background and Problem Formulation
## 2. 背景与问题表述


### 2.1 The In-Context Policy Learning Problem
### 2.1 上下文策略学习问题


We consider a robot manipulation agent operating in episodic trials. At each timestep $t$ ,the agent observes images ${o}_{t}$ and proprioceptive state ${s}_{t}$ ,and produces an action ${a}_{t} \in  \mathcal{A}$ . A trial terminates with outcome $\tau  = \left\{  {r,T,{\phi }_{1},\ldots ,{\phi }_{k}}\right\}$ where $r$ is the scalar reward, $T$ is the duration,and ${\phi }_{i}$ are structured diagnostic signals (phase log, minimum distance to target, oscillation flag).
我们考虑一个在情景试验中运行的机器人操作智能体。在每个时间步 $t$，智能体观察图像 ${o}_{t}$ 和本体感觉状态 ${s}_{t}$，并产生一个动作 ${a}_{t} \in  \mathcal{A}$。试验以结果 $\tau  = \left\{  {r,T,{\phi }_{1},\ldots ,{\phi }_{k}}\right\}$ 终止，其中 $r$ 是标量奖励，$T$ 是持续时间，${\phi }_{i}$ 是结构化诊断信号（阶段日志、到目标的最小距离、振荡标志）。


The in-context policy learning problem is: given the history of $n$ past outcomes ${H}_{n} = \left\{  {{\tau }_{1},\ldots ,{\tau }_{n}}\right\}$ and a set of key-frame images ${\mathcal{I}}_{n}$ ,produce an improved policy ${\pi }_{n + 1}$ without updating any model weights. Prior approaches differ in how they represent $\pi$ :
情景内策略学习问题定义如下：给定 $n$ 次过去试验结果 ${H}_{n} = \left\{  {{\tau }_{1},\ldots ,{\tau }_{n}}\right\}$ 的历史记录和一组关键帧图像 ${\mathcal{I}}_{n}$，在不更新任何模型权重的情况下产生一个改进的策略 ${\pi }_{n + 1}$。现有方法在如何表示 $\pi$ 上有所不同：


- Parameter update: $\pi$ is a fixed architecture; in-context improvement adjusts a parameter vector $\theta$ [24].
- 参数更新：$\pi$ 是固定架构；情景内改进调整参数向量 $\theta$ [24]。


- Skill selection: $\pi$ selects from a fixed skill library $\mathcal{S}$ ; in-context improvement changes the selection policy [1, 11].
- 技能选择：$\pi$ 从固定技能库 $\mathcal{S}$ 中进行选择；情景内改进改变选择策略 [1, 11]。


- Reward synthesis: $\pi$ is trained by RL; in-context improvement rewrites the reward function [16].
- 奖励合成：$\pi$ 通过强化学习训练；情景内改进重写奖励函数 [16]。


- Code synthesis (AOR): $\pi$ is an executable Python class; in-context improvement rewrites the entire implementation.
- 代码合成 (AOR)：$\pi$ 是一个可执行的 Python 类；情景内改进重写整个实现。


The code synthesis approach is the most expressive: it can represent all of the above plus arbitrary algorithmic changes, new phases, and new geometric computations. Its risk is that the hypothesis space is so large that an LLM may produce incorrect or unsafe code. AOR addresses this with a compilation sandbox, action clamping, and a fallback to the previous working controller.
代码合成方法最具表达力：它不仅能表示上述所有内容，还能进行任意算法更改、添加新阶段以及执行新的几何计算。其风险在于假设空间过大，可能导致大语言模型生成错误或不安全的代码。AOR 通过编译沙箱、动作钳制以及回退到上一个有效控制器来解决这一问题。


## 3. Related Work
## 3. 相关工作


### 3.1 Verbal Self-Reflection for Agents
### 3.1 智能体的语言自我反思


Reflexion [24] is the closest antecedent to AOR. It introduced the principle of verbal reinforcement: after each trial, the agent produces a natural-language reflection stored in a persistent memory buffer, which is prepended to the context for the next trial. Reflexion demonstrated 22% improvement on AlfWorld and 11% on HumanEval. However, Reflexion was designed for text-based environments (symbolic planning, code generation) where observations are structured and actions are discrete. The reflections modify strategy descriptions in natural language, not executable motor-control code.
Reflexion [24] 是 AOR 最接近的前身。它引入了语言强化原则：在每次试验后，智能体产生一段自然语言反思并存储在持久内存缓冲区中，该反思会被预置到下一次试验的上下文中。Reflexion 在 AlfWorld 上实现了 22% 的提升，在 HumanEval 上实现了 11% 的提升。然而，Reflexion 是为文本环境（符号规划、代码生成）设计的，其中观察是结构化的，动作是离散的。其反思修改的是自然语言中的策略描述，而非可执行的电机控制代码。


AOR extends Reflexion to physical manipulation in three ways: (i) observations include raw RGB-D images requiring visual reasoning; (ii) the policy is executable motor-control code, not a language description; (iii) failure modes involve continuous kinematics and sensor geometry, not just logical errors. The key insight from Reflexion that AOR preserves and generalises is that verbalising failure modes in natural language before rewriting policy is more effective than blind parameter search.
AOR 从三个方面将 Reflexion 扩展到了物理操作领域：(i) 观察包含需要视觉推理的原始 RGB-D 图像；(ii) 策略是可执行的电机控制代码，而非语言描述；(iii) 故障模式涉及连续运动学和传感器几何结构，而不仅仅是逻辑错误。AOR 保留并推广了 Reflexion 的核心洞察：在重写策略前用自然语言将故障模式口语化，比盲目的参数搜索更有效。


Hong et al. [10] propose Reflective Test-Time Planning for embodied LLMs, combining reflection-in-action (preexecution candidate generation), reflection-on-action (post-execution updates), and retrospective reflection (long-horizon credit assignment). Their work is the most direct embodied parallel to Reflexion and demonstrates strong improvements on long-horizon household tasks and MuJoCo environments. Feng et al. [8] extend this line with a VLM-based framework that imagines future world states to guide action selection across multi-stage manipulation tasks, outperforming MCTS and commercial VLMs on long-horizon benchmarks. The key distinction from AOR is the level of abstraction: both reflective planning approaches operate at the level of sub-goal planning-deciding which pre-built skill to invoke next-while AOR operates at the level of skill implementation, rewriting the motor-control code that executes a skill. The approaches are thus complementary rather than competing: one could use reflective planning for sub-goal selection while AOR handles low-level controller adaptation.
Hong 等人 [10] 为具身大语言模型提出了反射式测试时规划，结合了行动中反射（执行前候选生成）、行动后反射（执行后更新）和回顾性反射（长程信用分配）。他们的工作是具身领域对 Reflexion 最直接的并行研究，并在长程家庭任务和 MuJoCo 环境中表现出显著提升。Feng 等人 [8] 通过一个基于视觉语言模型的框架扩展了这一思路，该框架通过想象未来世界状态来指导多阶段操作任务中的动作选择，在长程基准测试中优于 MCTS 和商业视觉语言模型。与 AOR 的关键区别在于抽象层级：两种反射式规划方法均在子目标规划层面运行——即决定下一步调用哪种预构建技能——而 AOR 在技能实现层面运行，即重写执行技能的电机控制代码。因此，这些方法是互补而非竞争的：可以利用反射式规划进行子目标选择，同时由 AOR 处理底层控制器适配。


Self-Refine [18] generalises the reflection idea to arbitrary generation tasks: the same LLM produces output, critiques it, then refines it iteratively, improving code optimisation, maths, and dialogue quality. The generate-feedback-refine loop in AOR is structurally identical, but grounded in a physical domain: the "feedback" is visual and kinematic evidence from an episode rollout, not LLM self-critique, and the "output" is executable motor-control code, not free-form text.
Self-Refine [18] 将反思思想推广到任意生成任务：同一大语言模型生成输出、进行批判，然后迭代优化，从而提升代码优化、数学和对话质量。AOR 中的“生成-反馈-精炼”循环在结构上是相同的，但扎根于物理领域：“反馈”是来自试验过程的视觉和运动学证据，而非大语言模型的自我批判；“输出”是可执行的电机控制代码，而非自由格式的文本。


ReAct [32] interleaved reasoning traces with environment interactions, achieving 34% absolute gains over RL/imitation baselines on AlfWorld with 1-2 in-context examples. Ada-Planner [26] added in-plan and out-of-plan revision strategies, reducing samples required by $2 \times$ . These works demonstrate strong in-context planning but, like Reflexion, operate in symbolic environments with discrete action spaces.
ReAct [32] 将推理轨迹与环境交互交织在一起，在 AlfWorld 上仅使用 1-2 个上下文示例，就比强化学习/模仿学习基线实现了 34% 的绝对增益。Ada-Planner [26] 增加了规划内和规划外的修订策略，减少了 $2 \times$ 所需的样本量。这些工作展示了强大的上下文规划能力，但与 Reflexion 一样，它们是在具有离散动作空间的符号环境中运行的。


### 3.2 Code Generation for Robot Control
### 3.2 机器人控制的代码生成


Code as Policies [15] demonstrated that an LLM can generate hierarchically structured robot control programs from natural language commands. Given a task description and a library of perception primitives, the LLM produces Python code that calls these primitives to implement the task. The key distinction from AOR: Code as Policies is one-shot-it generates code once and executes it. There is no closed-loop improvement from episode outcomes, no visual evidence of failure, and no iterative refinement. AOR adds the critical feedback loop: observe what happened, reason about why, rewrite.
Code as Policies [15] 证明了 LLM 可以根据自然语言指令生成分层结构的机器人控制程序。给定任务描述和感知原语库，LLM 会生成调用这些原语的 Python 代码来实现任务。与 AOR 的关键区别在于：Code as Policies 是单次执行的——它生成一次代码并执行。它没有基于剧集结果的闭环改进，没有失败的视觉证据，也没有迭代优化。AOR 增加了关键的反馈循环：观察发生了什么，推理原因，然后重写。


ProgPrompt [25] used Pythonic program specifications to ground LLM task plans in available robot actions, achieving state-of-the-art on VirtualHome. Like Code as Policies, it is one-shot at the plan level, though the program representation is more structured.
ProgPrompt [25] 使用 Python 程序规范将 LLM 任务规划与可用的机器人动作对齐，在 VirtualHome 上达到了最先进水平。与 Code as Policies 一样，它在规划层面是单次执行的，尽管其程序表示更为结构化。


SayPlan [21] and related works use LLMs to generate scene-graph-grounded plans from natural language. These approaches focus on what to do (task planning), not how to do it (motor-control implementation). AOR targets the latter.
SayPlan [21] 及相关工作利用 LLM 从自然语言生成基于场景图的规划。这些方法侧重于做什么（任务规划），而不是如何做（运动控制实现）。AOR 则针对后者。


ReKep [12] uses VLMs to express manipulation constraints as Python functions over tracked 3D keypoints, enabling zero-shot multi-stage and bimanual tasks from a single natural-language instruction. This is the closest prior work to AOR's representation: both write Python code that reasons over visual observations at runtime. The key distinction: ReKep generates constraint code once from a scene description; it does not improve that code across episodes. AOR explicitly accumulates failure evidence across trials and rewrites the code in response, enabling systematic correction of errors that are invisible in a single observation.
ReKep [12] 使用 VLM 将操作约束表示为基于跟踪 3D 关键点的 Python 函数，从而能够仅凭一条自然语言指令实现零样本多阶段和双臂任务。这是与 AOR 表示法最接近的先前工作：两者都编写在运行时对视觉观察进行推理的 Python 代码。关键区别在于：ReKep 从场景描述中一次性生成约束代码；它不会跨剧集改进代码。AOR 明确地跨试验积累失败证据并据此重写代码，从而能够系统地纠正单次观察中无法发现的错误。


### 3.3 LLM-Generated Reward Functions and Fitness Crite- ria
### 3.3 LLM 生成的奖励函数与适应度准则


Eureka [16] used GPT-4 to generate reward function code, optimised over many RL training runs via an evolutionary outer loop, achieving human-expert-level reward design on 83% of 29 RL environments including dexterous tasks such as pen spinning. DrEureka [17] extended this to automate sim-to-real domain randomisation. Language to Rewards [33] translates natural-language task descriptions into reward function parameters for MuJoCo locomotion and manipulation controllers, bridging semantic instruction and low-level opti-misation without manual reward engineering. RLCD [31] and related works use LLM feedback as a substitute for human reward labelling.
Eureka [16] 使用 GPT-4 生成奖励函数代码，并通过进化外循环在多次强化学习训练运行中进行优化，在包括转笔等灵巧任务在内的 29 个强化学习环境中的 83% 上达到了人类专家级的奖励设计水平。DrEureka [17] 将此扩展到自动化模拟到现实的域随机化。Language to Rewards [33] 将自然语言任务描述转换为 MuJoCo 运动和操作控制器的奖励函数参数，在无需手动奖励工程的情况下架起了语义指令与底层优化之间的桥梁。RLCD [31] 及相关工作使用 LLM 反馈作为人类奖励标注的替代方案。


The key distinction from AOR: Eureka-style methods require many full RL training runs as the inner loop. Each outer-loop LLM call is followed by thousands of gradient updates. AOR requires only episode execution-each LLM call is followed by a single trial. This makes AOR substantially cheaper and faster for rapid task adaptation, at the cost of lower peak performance on tasks that benefit from large-scale RL.
与 AOR 的关键区别在于：Eureka 类方法需要多次完整的强化学习训练运行作为内循环。外循环中每次调用 LLM 后都会进行数千次梯度更新。AOR 仅需要剧集执行——每次调用 LLM 后仅进行单次试验。这使得 AOR 在快速任务适应方面成本更低、速度更快，代价是在那些受益于大规模强化学习的任务上峰值性能较低。


The two approaches are complementary: Eureka is optimal when RL training is feasible and the goal is maximum performance; AOR is optimal when fast adaptation and interpretability matter more than peak performance.
这两种方法是互补的：当强化学习训练可行且目标是追求最高性能时，Eureka 是最优的；当快速适应和可解释性比峰值性能更重要时，AOR 是最优的。


### 3.4 LLM-Based Optimisation
### 3.4 基于 LLM 的优化


A growing body of work uses LLMs as black-box optimis-ers: rather than computing a gradient, the LLM is shown a history of (solution, score) pairs and asked to propose a better solution.
越来越多的工作开始将 LLM 用作黑盒优化器：LLM 不再计算梯度，而是被展示（解决方案，分数）对的历史记录，并被要求提出更好的解决方案。


OPRO [30] formalised this idea directly: a meta-prompt contains past solutions sorted by score, and the LLM is asked to generate new candidates to maximise the objective. On prompt engineering and linear regression tasks OPRO matched or exceeded human-crafted solutions using only natural-language feedback. FunSearch [23] applied the same principle to combinatorial mathematics: an evolutionary outer loop selects high-scoring programs and feeds them back to the LLM as context, discovering new results in extremal graph theory and bin-packing. EvoPrompting [5] used LLMs to propose neural architecture mutations, combining evolutionary search with in-context program synthesis.
OPRO [30] 直接形式化了这一思想：元提示词包含按分数排序的过去解决方案，LLM 被要求生成新的候选方案以最大化目标。在提示词工程和线性回归任务中，OPRO 仅使用自然语言反馈就达到或超过了人工设计的解决方案。FunSearch [23] 将同样的原理应用于组合数学：进化外循环选择高分程序并将其作为上下文反馈给 LLM，从而在极值图论和装箱问题中发现了新结果。EvoPrompting [5] 使用 LLM 提出神经架构变异，将进化搜索与上下文程序合成相结合。


AOR is an instance of LLM-based optimisation where the objective is episode success rate, the solution space is Python controller code, and the "score" is a rich multimodal signal — key-frame images, reward traces, and phase-transition logs - rather than a scalar. The critical difference from OPRO and FunSearch is that AOR's feedback is visual and causal: the LLM can see where the arm was at the moment of failure and reason about the physical cause, not merely observe that the score was low. This richer feedback enables targeted single-step improvements (e.g. fixing a sign error in the back-projection formula) that pure score-based optimisation would require many trials to discover by coincidence.
AOR 是基于大语言模型（LLM）优化的一种实例，其目标是提升任务成功率，解空间为 Python 控制器代码，而“评分”不再是标量，而是丰富的多模态信号——包括关键帧图像、奖励轨迹和阶段转换日志。与 OPRO 和 FunSearch 的关键区别在于，AOR 的反馈是视觉化且具有因果关系的：LLM 不仅能观察到分数较低，还能看到机械臂在失败瞬间的位置，并推断出物理层面的原因。这种更丰富的反馈实现了针对性的单步改进（例如修复反投影公式中的符号错误），而纯粹基于分数的优化则需要多次试验才能偶然发现这些改进。


### 3.5 Grounded Planning with LLMs
### 3.5 基于 LLM 的接地规划


SayCan [1] grounded high-level LLM planning with a learned value function scoring which robot skills are executable in the current scene. PaLM-SayCan achieved 74% execution success on long-horizon manipulation tasks. The key limitation is that SayCan requires pre-trained low-level skills; it cannot improve those skills through reflection.
SayCan [1] 通过一个学习到的价值函数，将高级大语言模型（LLM）规划与当前场景中可执行的机器人技能评分相结合。PaLM - SayCan 在长周期操作任务上实现了 74% 的执行成功率。其主要局限性在于，SayCan 需要预训练的低级技能；它无法通过反思来改进这些技能。


Inner Monologue [11] continuously injected environment feedback as a running "inner monologue"—passive scene descriptions and success detection—into the LLM prompt, enabling closed-loop replanning. Inner Monologue improved on SayCan by allowing the LLM to detect and recover from execution failures, but its feedback is high-level ("the object was not grasped"), not diagnostic ("the arm descended 6 cm above the object because the depth centroid is on the visible surface, not the centre"). AOR's key contribution over Inner Monologue is the depth of diagnosis: by reasoning over images and controller code simultaneously, the LLM can pinpoint root causes at the level of individual lines of code.
《内心独白》[11] 持续将环境反馈作为持续的 “内心独白”（被动场景描述和成功检测）注入大语言模型提示中，从而实现闭环重新规划。《内心独白》对《SayCan》进行了改进，使大语言模型能够检测执行失败并从中恢复，但其反馈是高层次的（“物体未被抓取”），而非诊断性的（“机械臂在物体上方下降了 6 厘米，因为深度质心在可见表面，而非中心”）。与《内心独白》相比，AOR 的关键贡献在于诊断深度：通过同时对图像和控制器代码进行推理，大语言模型能够在代码的单行层面找出根本原因。


GPT-4V for Robotics [28] uses multimodal LLMs to analyse human demonstration videos and synthesise robot programs. This is complementary to AOR: it provides a rich warm-start policy from demonstrations; AOR then refines it through self-improvement.
用于机器人技术的 GPT - 4V [28] 使用多模态大语言模型来分析人类演示视频并合成机器人程序。这与 AOR 互补：它从演示中提供了丰富的热启动策略；然后 AOR 通过自我改进对其进行优化。


### 3.6 Foundation Models and VLAs for Manipulation
### 3.6 用于操作的基础模型与 VLA


The dominant paradigm for generalised robot manipulation trains vision-language-action (VLA) models jointly on internet-scale data and robot trajectories. RT-2 [4] co-fine-tuned a large VLM on robot data, achieving emergent generalisation. PaLM-E [7] injected continuous sensor modalities into a 562B-parameter language model. OpenVLA [14] provided an open-source 7B-parameter alternative that outperforms RT-2-X (55B) by 16.5% with LoRA fine-tuning. ${\pi }_{0}$ [3] combined a VLM backbone with a flow-matching action expert for dexterous manipulation. Octo [9] introduced the first open-source generalist policy trained on 800k episodes from Open X-Embodiment, supporting both language and goal-image conditioning with strong cross-embodiment transfer. The Open X-Embodiment [20] dataset and Gato [22] established the multi-task, multi-embodiment training paradigm. Most recently, dual-system architectures such as GR00T N1 [19] mirror cognitive fast/slow thinking: a slow VLM (System 2) reasons about instructions and scene context while a fast visuomotor policy (System 1) produces real-time actions. This two-system structure echoes AOR's two-timescale design, but the timescale separation in GR00T N1 operates within a single inference pass, whereas AOR's slow loop operates between episodes, accumulating experience to rewrite the fast controller.
通用机器人操作的主流范式是在互联网规模数据和机器人轨迹上联合训练视觉-语言-动作（VLA）模型。RT-2 [4] 将大型视觉语言模型（VLM）在机器人数据上进行联合微调，实现了涌现式泛化。PaLM-E [7] 将连续传感器模态注入到一个 5620 亿参数的语言模型中。OpenVLA [14] 提供了一个开源的 70 亿参数替代方案，通过 LoRA 微调，其性能比 RT-2-X (55B) 提升了 16.5%。${\pi }_{0}$ [3] 将 VLM 主干与流匹配动作专家相结合，用于灵巧操作。Octo [9] 推出了首个基于 Open X-Embodiment 80 万条轨迹训练的开源通用策略，支持语言和目标图像条件化，并具备强大的跨具身迁移能力。Open X-Embodiment [20] 数据集和 Gato [22] 确立了多任务、多具身的训练范式。最近，诸如 GR00T N1 [19] 等双系统架构模拟了认知中的快/慢思维：慢速 VLM（系统 2）负责推理指令和场景上下文，而快速视觉运动策略（系统 1）负责生成实时动作。这种双系统结构与 AOR 的双时间尺度设计相呼应，但 GR00T N1 的时间尺度分离是在单次推理过程中进行的，而 AOR 的慢速循环则运行在片段之间，通过积累经验来重写快速控制器。


AOR occupies a fundamentally different position from VLA models along several axes (Table 1). VLA models optimise for breadth: they generalise across hundreds of tasks through pretraining. AOR optimises for depth and interpretability: it achieves high performance on individual tasks through targeted reasoning, with a policy that can be read, audited, and modified by a human. The two are complementary: AOR-style reflection could provide targeted debugging when a VLA model fails on a specific deployment configuration without requiring full retraining.
AOR 在多个维度上与 VLA 模型处于截然不同的位置（表 1）。VLA 模型优化的是广度：通过预训练在数百个任务中实现泛化。AOR 优化的是深度和可解释性：通过针对性推理在单个任务上实现高性能，且策略可被人类阅读、审计和修改。两者是互补的：当 VLA 模型在特定部署配置下失败时，AOR 式的反思可以提供针对性的调试，而无需进行完全重训练。


Diffusion Policy [6] and ACT [34] represent the state of the art in data-driven visuomotor policy learning. Diffusion Policy outperforms prior methods by 46.9% on 12 tasks using multi-modal action distributions; ACT achieves 80-90% on bimanual tasks from 10 minutes of demonstrations. These establish an important comparison: given demonstrations, data-driven methods outperform AOR. The contribution of AOR is performance without any demonstrations.
Diffusion Policy [6] 和 ACT [34] 代表了数据驱动视觉运动策略学习的最新水平。Diffusion Policy 通过多模态动作分布在 12 项任务上比先前方法提升了 46.9%；ACT 在 10 分钟演示数据下于双臂任务中达到了 80-90% 的成功率。这些方法确立了一个重要的对比基准：在有演示数据的情况下，数据驱动方法优于 AOR。AOR 的贡献在于无需任何演示即可实现高性能。


### 3.7 In-Context Learning and Adaptation in Robotics
### 3.7 机器人学中的上下文学习与自适应


Instant Policy [27] demonstrated in-context imitation learning from 1-2 demonstrations via graph diffusion over scene graphs, enabling zero-shot generalisation to new tasks without weight updates. This is the closest gradient-free test-time adaptation baseline to AOR; the key distinction is that Instant Policy adapts to a new task from a demonstration, whereas AOR improves on the same task through its own failure history without any demonstrations.
即时策略 [27] 通过场景图上的图扩散，展示了从 1 - 2 次演示中进行上下文内模仿学习的能力，能够在不更新权重的情况下零样本泛化到新任务。这是与 AOR 最接近的无梯度测试时自适应基线；关键区别在于，即时策略是从演示中适应新任务，而 AOR 则是通过自身的失败历史在没有任何演示的情况下改进同一任务。


RoboAgent [2] achieved generalisation across 38 tasks from 7,500 demonstrations by combining semantic augmentation with multi-task action-chunking. These approaches remain fundamentally demonstration-driven; AOR requires no demonstrations.
RoboAgent [2]通过将语义增强与多任务动作分块相结合，在7500次演示中实现了38项任务的泛化。这些方法从根本上仍然依赖演示；而AOR无需演示。


The approach most structurally similar to AOR at the code level is the concurrent work of Kagaya et al. on memory-augmented code adaptation [13], which maintains a library of successful control programs from prior environments and retrieves and adapts them as in-context examples for new settings. The key distinction is directionality: that work retrieves successes to prime new tasks, whereas AOR diagnoses failures- using visual keyframes as evidence-to iteratively correct the same task's controller across episodes.
在代码层面上，与 AOR 结构最相似的方法是 Kagaya 等人关于内存增强代码适配的同期研究 [13]，该研究维护了一个来自先前环境的成功控制程序库，并将其作为上下文示例检索和适配到新环境中。关键区别在于方向：该研究检索成功案例以启动新任务，而 AOR 则利用视觉关键帧作为证据诊断失败情况，以在多个回合中迭代修正同一任务的控制器。


Closest in spirit to AOR otherwise is Eureka's use of LLM-generated code with a feedback loop-but as argued above, the feedback loop in Eureka involves RL training, not episode-level trial and error. To our knowledge, AOR is the first system to demonstrate iterative controller implementation improvement for robot manipulation via LLM reflection over visual evidence, without any gradient updates.
在理念上与 AOR 最接近的是 Eureka 利用 LLM 生成代码并结合反馈循环的方法，但如上所述，Eureka 的反馈循环涉及强化学习训练，而非回合级的试错。据我们所知，AOR 是首个通过 LLM 对视觉证据进行反思，在无需任何梯度更新的情况下，实现机器人操作控制器迭代改进的系统。


### 4.The AOR Framework
### 4. AOR 框架


AOR instantiates a two-timescale loop illustrated in Figure 1. The fast loop runs within an episode at control frequency; the slow loop runs between episodes at reasoning frequency. This decomposition is a deliberate design choice motivated by three considerations.
AOR 实例化了一个如图 1 所示的双时间尺度循环。快循环在回合内以控制频率运行；慢循环在回合间以推理频率运行。这种分解是基于三点考虑的刻意设计。


Separation of concerns. Physical control requires low-latency computation with deterministic timing guarantees. LLM inference is slow, expensive, and non-deterministic. Keeping the LLM strictly between episodes preserves real-time control properties while maximising the quality of the between-episode reasoning.
关注点分离。物理控制需要低延迟计算和确定性的时间保证。LLM 推理速度慢、成本高且具有非确定性。将 LLM 严格置于回合之间，既保留了实时控制特性，又最大化了回合间推理的质量。


Episode-level abstraction. Single-timestep observations are too noisy for reliable LLM reasoning about manipulation failures. Episode-level summaries (phase logs, key frames, reward trajectories) abstract away within-episode noise and expose the systematic patterns that the LLM needs to reason about.
回合级抽象。单时间步的观测对于 LLM 判断操作失败而言噪声过大。回合级摘要（阶段日志、关键帧、奖励轨迹）过滤了回合内的噪声，并揭示了 LLM 进行推理所需的系统性模式。


Code as the interface. The Python Controller class is a natural interface between LLM reasoning and robot execution: it is human-readable (supporting diagnosis), machine-executable (supporting deployment), and compositionally modifiable (supporting targeted rewrites).
代码作为接口。Python 控制器类是 LLM 推理与机器人执行之间的天然接口：它既易于人类阅读（支持诊断），又可由机器执行（支持部署），且具有组合可修改性（支持针对性重写）。

### 4.1 The Four Components
### 4.1 四大组件


Vision pipeline: Converts RGB-D images into a feature dict ${\mathbf{f}}_{t} = \{$ object_pos,eef_pos,detected,... $\}$ at each timestep. The vision pipeline is deliberately separate from the controller so that the LLM can reason about and rewrite them independently. The instantiation uses HSV colour-based segmentation, with the camera convention corrections described in Section 5.
视觉流水线：在每个时间步将 RGB-D 图像转换为特征字典 ${\mathbf{f}}_{t} = \{$ object_pos,eef_pos,detected,... $\}$。视觉流水线特意与控制器分离，以便 LLM 能够独立地对其进行推理和重写。该实例化方案使用基于 HSV 的颜色分割，并采用了第 5 节中描述的相机惯例校正。


Controller: A Python class with reset () and get_action (features) methods, safety-clamped via a compilation sandbox. The controller can implement any strategy expressible in Python with NumPy: phase-based state machines, PID loops, waypoint sequences, geometric computations. This expressiveness is the key advantage of code synthesis over parameter-tuning or skill-selection approaches.
控制器：一个包含 reset() 和 get_action(features) 方法的 Python 类，通过编译沙箱进行安全限制。控制器可以实现任何能用 Python 和 NumPy 表示的策略：基于阶段的状态机、PID 循环、路径点序列、几何计算等。这种表达能力是代码合成相较于参数调整或技能选择方法的关键优势。


Episodic memory: Stores structured episode outcomes (reward, step count, phase-transition log, minimum EEF-to-object distance, oscillation flag) and a small set of key-frame images captured at phase transitions and at regular intervals. The memory is persistent across sessions.
情景记忆：存储结构化的任务结果（奖励、步数、阶段转换日志、末端执行器到物体的最小距离、振荡标志）以及在阶段转换时和定期捕获的一小组关键帧图像。该记忆在不同会话间保持持久。


Multimodal LLM agent: Receives the full memory context plus the current controller source code and produces a new controller class. The agent is prompted to answer three questions before producing code: (1) What was the dominant failure mode? (2) Was the root cause in vision, controller logic, or parameters? (3) What is the single most impactful change? This structured reflection is directly inspired by Reflexion's [24] verbal reinforcement principle but instantiated for code synthesis.
多模态 LLM 智能体：接收完整的记忆上下文以及当前的控制器源代码，并生成一个新的控制器类。智能体在生成代码前会被提示回答三个问题：(1) 主要的故障模式是什么？(2) 根本原因是出在视觉、控制器逻辑还是参数上？(3) 最具影响力的单一改进是什么？这种结构化反思直接受到 Reflexion [24] 语言强化原则的启发，但将其应用于代码合成。


### 4.2 Safety and Stability Mechanisms
### 4.2 安全与稳定性机制


Several mechanisms prevent the code synthesis loop from producing unsafe or degenerate behaviour:
多种机制可防止代码合成循环产生不安全或退化的行为：


Compilation sandbox. All generated code is compiled and validated in an isolated namespace before execution. Any code that fails to compile, lacks the required interface, or raises exceptions at instantiation falls back to the previous working controller.
编译沙箱。所有生成的代码在执行前都会在隔离的命名空间中进行编译和验证。任何编译失败、缺少必要接口或在实例化时引发异常的代码，都会回退到上一个可用的控制器。


Action clamping. Every action returned by the controller is safety-clamped: $\left\lbrack  {-2,2}\right\rbrack$ for Cartesian deltas, $\left\lbrack  {-1,1}\right\rbrack$ for the gripper. Runtime exceptions per step produce a zero (safe-stop) action.
动作钳制。控制器返回的每个动作都会经过安全钳制：$\left\lbrack  {-2,2}\right\rbrack$ 用于笛卡尔增量，$\left\lbrack  {-1,1}\right\rbrack$ 用于夹爪。每一步的运行时异常都会产生零动作（安全停止）。


Bounded rewrites. The LLM is instructed to make targeted changes addressing the diagnosed failure mode, preserving the overall structure of the previous controller. Holistic rewrites that abandon all prior knowledge are discouraged. This is analogous to the bounded update principle in Reflexion.
有界重写。LLM 被指示针对诊断出的故障模式进行有针对性的修改，以保留前一个控制器的整体结构。不鼓励放弃所有先验知识的全面重写。这类似于 Reflexion 中的有界更新原则。

### 5.1 Vision Pipeline Design Choices
### 5.1 视觉流水线设计选择


#### 5.1.1 Colour segmentation and the camera convention prob- lem
#### 5.1.1 颜色分割与相机惯例问题


For tasks with known-colour objects, the agent's vision pipeline converged on HSV colour segmentation. A key insight discovered through AOR's reflexive loop concerns camera convention consistency. Robosuite uses IMAGE_CONVENTION = '`opengl'', meaning row 0 is at the image bottom and the camera uses $y$ -up, $z$ -back axes. The correct pinhole back-projection is:
对于已知颜色的物体，智能体的视觉流水线收敛于 HSV 颜色分割。通过 AOR 的自省循环发现的一个关键洞察涉及相机约定的统一性。Robosuite 使用 IMAGE_CONVENTION = 'opengl'，这意味着第 0 行位于图像底部，且相机使用 $y$ -向上、$z$ -向后的轴。正确的针孔反投影公式为：


$$
{\mathbf{p}}_{\text{ cam }} = {\left\lbrack  {x}_{p},{y}_{p}, - d,1\right\rbrack  }^{\top },\;{x}_{p} = \frac{\left( {u - {c}_{x}}\right) d}{{f}_{x}},\;{y}_{p} = \frac{\left( {v - {c}_{y}}\right) d}{{f}_{y}}.
$$



(1)



The standard OpenCV formula uses $- {y}_{p}$ ,producing 5- $8\mathrm{\;{cm}}$ errors. A further subtlety is that robosuite’s get_camera_extrinsic_matrix applies an OpenCV axis correction incompatible with OpenGL images; the world-to-camera transform must be computed directly from cam_xmat. These bugs are subtle, undocumented, and were autonomously discovered by the AOR reflexive loop—not by manual inspection (Section 6.2).
标准的 OpenCV 公式使用 $- {y}_{p}$，会产生 5- $8\mathrm{\;{cm}}$ 的误差。另一个微妙之处在于，robosuite 的 get_camera_extrinsic_matrix 应用了与 OpenGL 图像不兼容的 OpenCV 轴校正；世界坐标系到相机坐标系的变换必须直接从 cam_xmat 计算得出。这些错误非常隐蔽且缺乏文档记录，是由 AOR 自省循环自主发现的，而非通过人工检查（第 6.2 节）。


### 5.2 Controller Representation
### 5.2 控制器表示


The default controller template implements a sequential phase-based state machine: reach $\rightarrow$ descend $\rightarrow$ grasp $\rightarrow$ lift $\rightarrow  \{$ task-specific phases $\}$ . Phase transitions are threshold-based; each phase has a timeout fallback.
默认的控制器模板实现了一个基于阶段的顺序状态机：到达 $\rightarrow$、下降 $\rightarrow$、抓取 $\rightarrow$、提升 $\rightarrow  \{$、任务特定阶段 $\}$。阶段转换基于阈值；每个阶段都有超时回退机制。


Two design principles emerged as invariant across all instantiations through the agent's iterative reasoning:
通过智能体的迭代推理，有两种设计原则在所有实例化中表现为不变量：


Stationary grasp. During gripper closure, the EEF holds its position. Any downward pressing force destabilises the object. This is geometrically obvious but was not in the default controller; the LLM identified it as the root cause of grasp failure in the first Lift iteration.
静止抓取。在夹爪闭合期间，末端执行器（EEF）保持其位置。任何向下的压力都会使物体不稳定。这一点在几何上显而易见，但并未包含在默认控制器中；LLM 将其识别为首次 Lift 迭代中抓取失败的根本原因。


EMA smoothing. Actions are smoothed via ${\mathbf{a}}_{t} = \alpha {\widehat{\mathbf{a}}}_{t} + \; \left( {1 - \alpha }\right) {\mathbf{a}}_{t - 1}$ with $\alpha  \in  \left\lbrack  {{0.3},{0.5}}\right\rbrack$ . The appropriate $\alpha$ depends on object geometry: larger, more inertially stable objects tolerate higher $\alpha$ ; lighter objects require higher smoothing to maintain grip integrity.
EMA 平滑。动作通过 ${\mathbf{a}}_{t} = \alpha {\widehat{\mathbf{a}}}_{t} + \; \left( {1 - \alpha }\right) {\mathbf{a}}_{t - 1}$ 进行平滑处理，参数为 $\alpha  \in  \left\lbrack  {{0.3},{0.5}}\right\rbrack$。合适的 $\alpha$ 取决于物体几何形状：更大、惯性更稳定的物体可以容忍更高的 $\alpha$；较轻的物体则需要更高的平滑度以保持抓取的完整性。

Systematic failures: rapid convergence Lift converged to 100% in 3 LLM calls. 调用 1 - 物体颜色错误。第 0 集显示机械臂在 reach 阶段停滞了全部 1000 步。LLM 诊断出完全检测失败：初始的 detect_silver_can 例程（低饱和度 HSV 掩码）未检测到任何物体，因为在 agentview 摄像机中，罐子呈现为红色而非金属银色。视觉分支被重写为使用红色色调分割器。

调用 2 - 容器指示器干扰。在启用红色检测后，第 1 集显示机械臂向场景的容器侧下降，而非桌上的罐子。LLM 诊断结果为：detect_red_object 对帧内所有红色像素取平均值；robosuite 容器区域包含一个指示罐子目标槽位的小红色标记，其像素将质心向右拉。修复方法是切换到最大连通分量过滤（detect_red_can），将罐子（最大的红色区域）与较小的容器指示器隔离开来。

## 7. Discussion
## 7. 讨论


### 7.1 Code as the Right Representation for Reflective Learn- ing
### 7.1 代码是反射式学习的正确表征


The central question of this paper was whether making the full controller implementation the unit of LLM reasoning qualitatively changes in-context learning in physical manipulation. The evidence says yes.
本文的核心问题是：将完整的控制器实现作为大语言模型（LLM）推理的单元，是否会从本质上改变物理操作中的上下文学习。证据表明确实如此。


Consider the Stack task vision bugs. In a parameter-tuning framework (Reflexion-style), the LLM would observe that placement precision is poor and reduce thresholds or adjust gains. These adjustments cannot fix a sign error in the back-projection formula. In a skill-selection framework (SayCan, Inner Monologue), the skill implementations are fixed; the LLM cannot reach into them. In Code as Policies, there is no feedback loop to trigger re-examination. Only a code-synthesis framework with a closed loop and diagnostic prompts can identify "the y-coordinate in the back-projection has the wrong sign" and rewrite it. This is a fundamental capability advantage, not just a quantitative improvement.
以 Stack 任务中的视觉错误为例。在参数调整框架（Reflexion 风格）中，LLM 会观察到放置精度较差，从而降低阈值或调整增益。这些调整无法修复反投影公式中的符号错误。在技能选择框架（SayCan、Inner Monologue）中，技能实现是固定的，LLM 无法对其进行干预。在“代码即策略”（Code as Policies）中，没有触发重新检查的反馈回路。只有具备闭环和诊断提示的代码合成框架，才能识别出“反投影中的 y 坐标符号错误”并对其进行重写。这是一种根本性的能力优势，而不仅仅是量化改进。


The colour-pipeline diagnosis in PickPlaceCan makes the same point: the episode log showed the arm stuck in the reach phase for all 1000 steps, indicating a total detection failure. Recognising that the can rendered as red rather than silver, and that averaging all red pixels produced a centroid biased toward the bin's colour marker, required understanding what the segmentor was computing and why. No parameter tuning or skill selection can express this; it requires rewriting the object-localisation logic.
PickPlaceCan 中的颜色流水线诊断也说明了这一点：剧集日志显示机械臂在所有 1000 个步骤中都卡在到达阶段，表明检测完全失败。要识别出罐子被渲染为红色而非银色，且所有红色像素的平均值导致质心偏向垃圾桶的颜色标记，就需要理解分割器在计算什么以及原因。任何参数调整或技能选择都无法表达这一点；它需要重写物体定位逻辑。


### 7.2 Autonomous Perception Debugging
### 7.2 自主感知调试


The autonomous discovery of the camera convention bugs deserves explicit discussion, as it illustrates a form of debugging absent from conventional robot learning. In standard imitation or RL pipelines, vision bugs are invisible: the policy simply fails to train to high performance, and the attribution to vision vs. controller is unclear.
自主发现相机约定错误值得明确讨论，因为它展示了一种传统机器人学习中缺失的调试形式。在标准的模仿学习或强化学习流水线中，视觉错误是不可见的：策略只是无法训练到高性能，且难以归因于视觉还是控制器。


In AOR, the LLM has access to both the observation data (images showing where the arm went) and the implementation code (the back-projection formula). After Stack iteration 6, the LLM produced a diagnosis of the form: "the controller changes have improved convergence speed but the placement is consistently wrong by 5-8 cm-this is larger than any achievable parameter-tuning effect and the magnitude is consistent with a systematic back-projection error." This level of attribution—from symptom to code module to specific line— is only possible because the LLM can read and reason about the code implementation.
在 AOR 中，LLM 可以访问观测数据（显示机械臂位置的图像）和实现代码（反投影公式）。在 Stack 第 6 次迭代后，LLM 给出了如下诊断：“控制器更改提高了收敛速度，但放置位置始终有 5-8 厘米的偏差——这超出了任何参数调整所能达到的效果，且偏差幅度与系统性的反投影错误一致。”这种从症状到代码模块再到具体代码行的归因能力，只有在 LLM 能够阅读并推理代码实现时才有可能实现。


This suggests that code-synthesis reflexive agents may be useful not just as policy learners but as debugging tools for any vision-based robot system, even ones that ultimately use learned policies.
这表明，代码合成反射式智能体不仅可用作策略学习者，还可用作任何基于视觉的机器人系统的调试工具，即使是那些最终使用学习策略的系统。


### 7.3 Comparison to the Reflexion Family
### 7.3 与 Reflexion 系列的比较


The most precise comparison is with Reflexion [24] and Reflective Test-Time Planning [10]. All three operate in the in-context improvement paradigm; the differences are architectural.
最精确的比较是与 Reflexion [24] 和 Reflective Test-Time Planning [10] 进行的。这三者都在上下文内改进范式下运作；区别在于架构方面。


Reflexion modifies verbal strategies, which are then interpreted by a high-level LLM producing discrete actions. This is effective in symbolic environments but cannot produce the geometric computations (back-projection correction, bin geometry target correction) needed for precise manipulation.
Reflexion 修改的是语言策略，这些策略随后由产生离散动作的高层 LLM 进行解释。这在符号环境中很有效，但无法产生精确操作所需的几何计算（反投影校正、箱体几何目标校正）。


Reflective Test-Time Planning [10] operates at the sub-goal level: it decides which pre-built primitive to invoke next and how to adapt the plan when a sub-goal fails. The primitives themselves are fixed. AOR is complementary: one could use Reflective Test-Time Planning to select which task to execute while AOR learns how to execute it.
反射式测试时规划 [10] 在子目标层面运行：它决定下一步调用哪个预构建的原语，以及在子目标失败时如何调整计划。原语本身是固定的。AOR 与之互补：可以使用反射式测试时规划来选择执行的任务，而由 AOR 学习如何执行该任务。


The generalisable principle emerging from all three works is: in-context improvement is effective when the level of abstraction at which the LLM operates matches the level at which failures occur. Reflexion matches text-level planning failures; Reflective TTP matches sub-goal-level failures; AOR matches motor-control implementation failures.
从这三项工作中得出的通用原则是：当 LLM 操作的抽象级别与故障发生的级别相匹配时，上下文改进是有效的。Reflexion 匹配文本层面的规划故障；反射式 TTP 匹配子目标层面的故障；AOR 匹配电机控制实现层面的故障。


## 8. Limitations and Future Work
## 8. 局限性与未来工作


Simulation only. All results are from simulation. Real robots introduce sensing noise, actuation imprecision, and lighting variation that would require tighter uncertainty modelling. The techniques the agent introduced (colour-based segmentation, camera convention correction, grasp-retry mechanisms) are motivated by physical principles and should transfer to real hardware.
仅限仿真。所有结果均来自仿真环境。真实机器人引入的传感噪声、执行不精确性和光照变化将需要更严谨的不确定性建模。智能体引入的技术（基于颜色的分割、相机约定校正、抓取重试机制）均受物理原理驱动，应能迁移至真实硬件。


Sequential iteration. Each hypothesis is tested one trial at a time. Parallel hypothesis testing-generating multiple candidate controllers and evaluating them simultaneously-would reduce wall-clock time substantially.
顺序迭代。每个假设均逐次进行试验。并行假设测试——即同时生成并评估多个候选控制器——将大幅缩短实际耗时。


LLM-dependence. The quality of diagnosis depends on the LLM's capability to reason about geometry, code, and physics. As LLMs improve, AOR performance is expected to improve without any changes to the framework.
对大语言模型（LLM）的依赖。诊断质量取决于 LLM 对几何、代码和物理规律的推理能力。随着 LLM 的进步，AOR 的性能有望在无需修改框架的情况下得到提升。


Incomplete search. The Stack experiments reveal a shortcoming of the reflexion search: the agent correctly identified gripper contact with the target cube as the cause of residual failures but exhausted its iteration budget without finding a placement strategy to avoid it. The agent converged to a local optimum in controller space. Mechanisms such as explicit diversity pressure, restarts, or structured exploration of placement strategies are not present in the current AOR loop and would be needed to overcome this kind of search failure.
搜索不完整。Stack 实验揭示了反射式搜索的不足：智能体正确识别出机械臂与目标方块的接触是导致残留故障的原因，但在耗尽迭代预算前未能找到规避该问题的放置策略。智能体在控制器空间内收敛到了局部最优解。当前的 AOR 循环中缺乏显式的多样性压力、重启机制或对放置策略的结构化探索，而这些对于克服此类搜索失败至关重要。


Classical controller bias. Left unprompted, the agent converged on classical control techniques-HSV segmentation, phase-based state machines, PID-style gains-because these are well-represented in its training data and produce interpretable, debuggable code. The agent is not inherently restricted to classical methods: prompting it to use a specific technique (learned visual features, neural controllers, Kalman filtering) would likely cause it to adopt that approach instead. The choice of technique is a prompt-level decision, and more modern perception or control methods could be introduced simply by instructing the agent to use them. Exploring this prompt-steerable technique selection is a direct avenue for future work.
经典控制器偏好。在无提示的情况下，智能体会收敛于经典控制技术（如 HSV 分割、基于阶段的状态机、PID 风格增益），因为这些方法在其训练数据中表现良好，且能生成可解释、可调试的代码。智能体并非本质上受限于经典方法：通过提示其使用特定技术（如学习到的视觉特征、神经控制器、卡尔曼滤波），很可能会促使其采用相应方案。技术选择属于提示词层面的决策，只需指示智能体使用更现代的感知或控制方法即可引入。探索这种可通过提示引导的技术选择是未来工作的直接方向。


Future directions: (i) Prompting the agent to use modern perception methods (e.g. learned keypoint detectors or foundation-model visual features) in place of hand-coded HSV segmentation, to improve robustness to lighting variation and novel objects. (ii) Combining AOR with a VLA prior for warm-started adaptation. (iii) Using Reflective Test-Time Planning [10] for sub-goal selection while AOR handles low-level implementation adaptation. (iv) Real-robot deployment with appropriate uncertainty modelling. (v) Theoretical characterisation of when in-context policy improvement converges, in terms of observability and reproducibility of failure modes.
未来方向：（i）提示智能体使用现代感知方法（如学习到的关键点检测器或基础模型视觉特征）替代手写 HSV 分割，以提高对光照变化和新物体的鲁棒性。（ii）将 AOR 与 VLA 先验结合，实现热启动适应。（iii）在 AOR 处理底层实现适应的同时，利用反射式测试时规划 [10] 进行子目标选择。（iv）结合适当的不确定性建模进行真实机器人部署。（v）从可观测性和故障模式的可复现性角度，对上下文策略改进的收敛性进行理论表征。


## 9. Conclusion
## 9. 结论


This paper presented Act-Observe-Rewrite, a framework for in-context policy learning in robot manipulation that makes executable Python controller code the unit of LLM reasoning. The framework advances beyond prior work in verbal reflection (Reflexion, Reflective Test-Time Planning) by operating at the level of motor-control implementation rather than strategy description or sub-goal selection, and beyond prior work in robot code generation (Code as Policies, ProgPrompt) by closing the feedback loop with visual episode evidence.
本文提出了 Act-Observe-Rewrite (AOR)，这是一个用于机器人操作上下文策略学习的框架，它将可执行的 Python 控制器代码作为 LLM 推理的单元。该框架超越了以往在语言反射（Reflexion、反射式测试时规划）方面的研究，其操作层面深入到电机控制实现而非仅限于策略描述或子目标选择；同时也超越了以往机器人代码生成（Code as Policies、ProgPrompt）的研究，通过视觉片段证据闭环了反馈机制。


On Lift and PickPlaceCan, AOR converges to 100%: the agent finds and corrects systematic failures-vision biases, incorrect colour identification, centroid contamination by scene markers, coordinate convention errors-through iterative diagnosis and code rewriting. On Stack, it reaches 91% and stalls. Every residual failure involves gripper contact with cubeB during placement; the agent observed this in episode images and correctly named it as the cause. It did not, however, find a controller variant that avoids the contact, despite such variants likely existing. This is an honest result: the reflexion search converged to a local optimum and did not recover.
在 Lift 和 PickPlaceCan 任务上，AOR 收敛至 100%：智能体通过迭代诊断和代码重写，发现并纠正了系统性故障，如视觉偏差、颜色识别错误、场景标记导致的质心污染以及坐标约定错误。在 Stack 任务上，成功率达到 91% 后陷入停滞。所有残留故障均涉及放置过程中机械臂与 cubeB 的接触；智能体在片段图像中观察到了这一点并准确将其命名为故障原因。然而，尽管存在规避接触的控制器变体，智能体却未能找到。这是一个真实的结果：反射式搜索收敛到了局部最优解，未能恢复。


Collectively, these results establish that multimodal coding agents occupy a distinct, practically useful position in the robot learning design space: no demonstrations, no reward engineering, no gradient updates-yet high task-specific performance, full interpretability, and a systematic approach to understanding why a controller fails. As foundation models become more capable reasoners about geometry and physics, we expect this combination of verbal diagnosis and code synthesis to become an increasingly important tool for deploying physical AI systems in the real world.
总而言之，这些结果确立了多模态编码智能体在机器人学习设计空间中占据着独特且实用的位置：无需演示、无需奖励工程、无需梯度更新，却能实现高任务特定性能、完全可解释性，并提供了一种理解控制器为何失效的系统化方法。随着基础模型在几何和物理推理能力上的增强，我们预期这种语言诊断与代码合成的结合，将成为在现实世界中部署物理 AI 系统日益重要的工具。

## Example 2: Vision pipeline bug isolated after 6 failed iter- ations (Stack, iteration 6)

Reasoning: "Six iterations of controller changes have produced zero improvement in placement accuracy. 置信度：0.85

## Where the agent does not do well
## 智能体的不足之处


Three systematic weaknesses emerged across the iteration sequence:
在整个迭代序列中，出现了三个系统性的弱点：


Bug introduction during refactoring. The v18 case above is the clearest example: a conceptually sound change (vary approach angle on retry) was implemented by modifying a shared parameter, inadvertently breaking placement. The agent does not "simulate" the effect of code changes on all phases—it reasons locally about the change it intends to make and misses side-effects on other uses of the same variable.
重构过程中引入错误。上述 v18 案例是最明显的例子：一个概念上合理的修改（在重试时改变接近角度）是通过修改共享参数来实现的，无意中破坏了放置功能。智能体不会“模拟”代码更改对所有阶段的影响——它仅针对打算进行的更改进行局部推理，从而忽略了对同一变量其他用途的副作用。


Stochastic vs. systematic confusion. In early Stack iterations, the agent repeatedly proposed threshold tightening and gain changes to improve grasp reliability. Many of these changes were neutral or slightly harmful because the grasp failures were stochastic (dependent on exact initial cube placement) rather than systematic. The agent required multiple iterations of observing the same failure mode with different controller parameters before concluding that controller changes were not the solution. A more efficient agent would distinguish stochastic variance from systematic bias earlier, for example by correlating failures with initial-state randomness across episodes.
随机性与系统性混淆。在早期的堆叠迭代中，智能体反复建议收紧阈值和更改增益以提高抓取可靠性。这些更改中有许多是中性或略有负面影响的，因为抓取失败是随机的（取决于立方体的初始放置位置），而非系统性的。智能体需要多次迭代观察同一失败模式在不同控制器参数下的表现，才能得出控制器更改并非解决方案的结论。一个更高效的智能体应该能更早地将随机方差与系统偏差区分开来，例如通过关联剧集间的初始状态随机性与失败情况。


Confidence miscalibration. The agent's confidence field frequently did not predict whether the proposed change would improve performance. High-confidence diagnoses (0.8+) sometimes led to regressions (v17: 6/10, below the 8/10 baseline), while low-confidence changes (0.5-0.6) sometimes produced the largest improvements (v16 retract ramp). The confidence field captures the agent's subjective certainty about the diagnosis but not its effect size or potential for side-effects.
置信度校准偏差。智能体的置信度字段经常无法预测所提议的更改是否会提升性能。高置信度的诊断（0.8+）有时会导致性能倒退（v17：6/10，低于 8/10 的基准），而低置信度的更改（0.5-0.6）有时却能带来最大的提升（v16 缩回斜坡）。置信度字段捕捉的是智能体对诊断结果的主观确定性，而非其影响程度或潜在的副作用。
