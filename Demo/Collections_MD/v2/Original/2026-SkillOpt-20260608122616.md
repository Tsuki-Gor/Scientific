Agent skills today are hand-crafted, generated one-shot, or evolved through loosely controlled self-revision - none of which behaves like a deep-learning optimizer for the skill, and none of which reliably improves over its starting point under feedback. We argue the skill should instead be trained as the external state of a frozen agent, with the same discipline that makes weight-space optimization reproducible. SKILLOPT is, to our knowledge, the first systematic controllable text-space optimizer for agent skills: a separate optimizer model turns scored rollouts into bounded add/delete/replace edits on a single skill document, and an edit is accepted only when it strictly improves a held-out validation score. A textual learning-rate budget, rejected-edit buffer, and epoch-wise slow/meta update make skill training stable while adding zero inference-time model calls at deployment. Across six benchmarks, seven target models, and three execution harnesses (direct chat, Codex, Claude Code), SKILLOPT is best or tied on all 52 evaluated (model, benchmark, harness) cells and beats every per-cell competitor among human, one-shot LLM, Trace2Skill, TextGrad, GEPA, and EvoSkill skills. On GPT-5.5 it lifts the average no-skill accuracy by +23.5 points in direct chat, by +24.8 inside the Codex agentic loop, and by +19.1 inside Claude Code. Transfer experiments further show that optimized skill artifacts retain value when moved across model scales, between Codex and Claude Code execution environments, and to a nearby math benchmark without further optimization.
如今的智能体技能要么是手工设计、一次性生成，要么通过宽松控制的自我修订演化而来——这些都不像针对技能的深度学习优化器，也都无法在反馈下可靠地优于初始点。我们认为，技能应当被训练为冻结智能体的外部状态，并采用与使权重空间优化可复现相同的规范。就我们所知，SKILLOPT 是首个面向智能体技能的系统化、可控文本空间优化器：一个独立的优化器模型将带评分的 rollout 转化为单个技能文档上的有界增删改编辑，且只有当编辑严格提升留出的验证分数时才会被接受。文本化学习率预算、被拒绝编辑缓冲区以及按 epoch 的慢/元更新，使技能训练稳定，同时在部署时增加零推理阶段模型调用。在六个基准、七个目标模型和三种执行框架（直接对话、Codex、Claude Code）上，SKILLOPT 在全部 52 个评估单元（模型、基准、框架）中表现最佳或持平，并在每个单元上都优于人类、one-shot LLM、Trace2Skill、TextGrad、GEPA 和 EvoSkill 技能中的各自竞争者。在 GPT-5.5 上，它将无技能平均准确率在直接对话中提升 +23.5 个百分点，在 Codex 智能体循环中提升 +24.8 个百分点，在 Claude Code 中提升 +19.1 个百分点。迁移实验进一步表明，优化后的技能产物在跨模型规模、在 Codex 与 Claude Code 执行环境之间迁移，以及迁移到附近的数学基准而无需进一步优化时，仍然保持价值。

## 1 Introduction
## 1 引言


Frontier language models are increasingly deployed as agents, from single-prompt callers to multistep execution harnesses with tools, files, and verifiers [1-4]. In such settings, domain adaptation is no longer only about model weights or prompts: it also requires improving the procedures by which the agent gathers evidence, calls tools, follows domain conventions, and formats outputs [5, 6]. Agent skills provide a natural interface for this procedural adaptation [7, 8]: a skill is a portable natural-language artifact that packages procedures, domain heuristics, tool policies, output constraints, and failure modes, letting a frozen agent adapt through external text.
前沿语言模型正越来越多地被部署为智能体：从单次提示调用者到带工具、文件和验证器的多步执行框架 [1-4]。在这种场景下，领域适配不再只是关于模型权重或提示：还需要改进智能体获取证据、调用工具、遵循领域约定以及格式化输出的流程 [5, 6]。智能体技能为这种程序性适配提供了自然接口 [7, 8]：技能是可移植的自然语言产物，用来打包流程、领域启发式、工具策略、输出约束和失败模式，使冻结的智能体能够通过外部文本进行适应。


If the recurring object of adaptation is the agent's procedure, the skill document itself should be trainable. Yet weight adaptation is often unavailable for closed frontier models and expensive for open ones, while manually written or one-shot skills are brittle under a target domain or harness. Recent systems convert execution experience into reusable textual artifacts- distilling trajectory lessons, refining skill folders via failure analysis, building domain-specific skill libraries, or optimizing prompts from trajectory feedback [9-13] - but leave open a more basic question: if skills are the adaptation layer, how should they be optimized? Our key idea is to treat skill editing as a controllable domain-adaptation process, with the skill document as the external state, an additional frontier model as the optimizer, and training-style controls over evidence, step size, validation, and update direction.
如果需要反复适配的是智能体的“过程”，那么技能文档本身就应该可以训练。然而，对于封闭的前沿模型，权重适配往往不可用；对开放模型则代价高昂；而手工编写或一次性（one-shot）的技能在目标领域或执行框架下往往又脆弱。近期系统将执行经验转化为可复用的文本产物——提炼轨迹经验教训，通过失败分析精炼技能文件夹，构建领域特定的技能库，或利用轨迹反馈来优化提示 [9-13]——但仍然留着一个更基础的问题：既然技能是适配层，那么应当如何对它们进行优化？我们的核心想法是将“技能编辑”视为一种可控的领域适配过程：以技能文档作为外部状态，再配一个额外的前沿模型作为优化器，并在训练风格的控制下，对证据、步长、验证以及更新方向进行调节。

We introduce SKILLOPT, a text-space optimizer for agent skills. Given a target domain, an initial skill, and the model being adapted, SKILLOPT repeatedly samples trajectory batches, analyzes successes and failures, and asks a frontier optimizer model to propose structured add/delete/replace edits. It then aggregates and ranks candidate edits under a textual learning-rate budget, applies a bounded update to the skill document, and evaluates the candidate skill on a held-out selection split before accepting it. Rejected edits are retained as negative feedback, while the epoch-wise slow/meta update preserves longer-horizon regularities. Figure 1 gives a schematic view of this loop. The deployed output is a compact best_skill.md file of roughly 300-2,000 tokens, with the adapted model and execution harness remaining fixed.
我们提出 SKILLOPT，这是一种用于智能体技能的文本空间优化器。给定目标领域、初始技能以及正在被适配的模型，SKILLOPT 会反复采样轨迹批次，分析成功与失败，并要求一个前沿优化器模型提出结构化的 add/delete/replace 编辑。随后，它在一个基于文本学习率预算下汇总并对候选编辑进行排序，对技能文档执行受限更新，并在一个隔离的选择划分上评估候选技能，然后再接受它。被拒绝的编辑会保留下来作为负反馈，而逐 epoch 的慢速/元（slow/meta）更新会维持更长时间跨度上的规律。图1 给出了该循环的示意。最终部署输出是一份紧凑的 best_skill.md 文件，约 300-2,000 个 token，并且被适配的模型与执行框架保持不变。


The deep-learning analogy is operational rather than decorative. Rollout and reflection batch sizes control the noise in the evidence used for each edit; the textual learning rate and schedule control how far one skill version is allowed to move from the previous one; the held-out gate plays the role of validation; and the epoch-wise slow/meta update acts like a momentum term, carrying stable editing directions across epochs. This stability is crucial: if consecutive skill revisions move too far or in inconsistent directions, rejected edits and previous accepted edits no longer provide a meaningful optimization history. With bounded, validation-gated updates, each revision remains close enough to the last one that later optimizer calls can learn from what helped, what failed, and what should be preserved.
该深度学习类比是“可操作”的，而不是为了装饰。回放（rollout）与反思（reflection）的批次规模决定了用于每次编辑的证据中的噪声大小；文本学习率及其调度决定单个技能版本允许从前一个版本移动多远；隔离的门控机制对应验证；而逐 epoch 的慢速/元更新就像动量项一样，在跨 epoch 中携带稳定的编辑方向。这种稳定性至关重要：如果连续的技能修订移动过远，或在不一致的方向上进行，那么被拒绝的编辑与先前被接受的编辑就不再能提供有意义的优化历史。在受限、且经过验证门控的更新下，每次修订都会与上一次保持足够接近，从而让后续优化器调用能从中学习：哪些有效、哪些失败、以及哪些应当被保留。


We conduct, to our knowledge, the first systematic study of skill optimization as a domain-adaptation training method for frontier agents. We evaluate SKILLOPT on six benchmarks covering QA, spreadsheets, documents, math, and embodied decision making, across seven target models from frontier-scale GPT to small-scale Qwen, and under three execution modes (direct chat, Codex harness, Claude Code harness). Out of 52 evaluated (model, benchmark, harness) cells, SKILLOPT is the best or tied-best measured method on all 52. With GPT-5.5 in direct chat, it lifts SearchQA from 77.7 to 87.3, SpreadsheetBench from 41.8 to 80.7, OfficeQA from 33.1 to 72.1, DocVQA from 78.8 to 91.2, LiveMathematicianBench from 37.6 to 66.9, and ALFWorld from 83.6 to 95.5 (a +23.5 point average gain over no skill), and it also beats the strongest per-cell baseline drawn from human-written, one-shot LLM, Trace2Skill, TextGrad, GEPA, and EvoSkill skills by +5.4 points on average. The same optimization interface is effective inside Codex-style and Claude Code-style execution loops, lifting GPT-5.5 by +24.8 and +19.1 points over no skill respectively, and outperforming EvoSkill by +14.0 and +3.2 points.
据我们所知，我们是首次系统性地研究将技能优化作为一种用于前沿智能体的领域适配训练方法。我们在六个基准上评估 SKILLOPT，覆盖 QA、表格（spreadsheets）、文档（documents）、数学与具身决策（embodied decision making），并在七个目标模型上进行实验，从前沿规模的 GPT 到小规模的 Qwen，同时在三种执行模式下测试（直接对话、Codex 框架、Claude Code 框架）。在共 52 个评估单元（模型-基准-框架）中，SKILLOPT 在所有 52 个单元上都是最优或并列最优。以 GPT-5.5 的直接对话为例，它把 SearchQA 从 77.7 提升到 87.3，把 SpreadsheetBench 从 41.8 提升到 80.7，把 OfficeQA 从 33.1 提升到 72.1，把 DocVQA 从 78.8 提升到 91.2，把 LiveMathematicianBench 从 37.6 提升到 66.9，把 ALFWorld 从 83.6 提升到 95.5（相对无技能基线的 +23.5 分平均增益），并且它也比从人类编写的一次性 LLM 技能（human-written, one-shot LLM）以及 Trace2Skill、TextGrad、GEPA、EvoSkill 中抽取的最强逐单元基线平均高出 +5.4 分。相同的优化接口同样在 Codex 风格与 Claude Code 风格的执行循环内有效：分别让 GPT-5.5 相对无技能提升 +24.8 和 +19.1 分，并且比 EvoSkill 分别高出 +14.0 和 +3.2 分。


The learned artifacts also transfer beyond the exact training setting. A SpreadsheetBench skill trained on GPT-5.4 improves every smaller GPT variant we test; a Codex-trained spreadsheet skill transfers to Claude Code with a +59.7 point gain; and an OlympiadBench skill yields positive gains on Omni-MATH [14]. These transfer results are important for the paper's application value: a skill can be optimized once, audited as text, and reused across related models, harnesses, or tasks without changing model weights. Our ablations explain why this works. Bounded textual learning outperforms uncontrolled rewriting, held-out gating prevents harmful proposals from accumulating, the rejected-step buffer converts failed edits into negative feedback, and the epoch-wise slow/meta update improves long-horizon refinement without bloating the deployed skill. Finally, per-benchmark case studies show that the learned skills remain compact (300-2,000 tokens after only 1-4 accepted edits), inspectable, and procedural rather than instance-specific.
所学得到的产物也会超越严格的训练设置进行迁移。在 GPT-5.4 上训练的 SpreadsheetBench 技能可提升我们测试的每个更小的 GPT 变体；在 Codex 上训练的表格技能迁移到 Claude Code 时带来 +59.7 分的提升；而 OlympiadBench 技能在 Omni-MATH [14] 上也带来正向收益。这些迁移结果对论文的应用价值很重要：一项技能可以只优化一次，以文本形式审计，并在不改变模型权重的情况下复用于相关模型、执行框架或任务。我们的消融实验解释了其有效原因。受限的文本学习优于不受控制的重写，保留集门控可防止有害提议累积，拒绝步骤缓冲区将失败编辑转化为负反馈，而按 epoch 的慢速/元更新则在不使部署后的技能膨胀的前提下改善长周期细化。最后，按基准的案例研究表明，所学技能仍然紧凑（仅 1-4 次接受编辑后为 300-2,000 个 token），可检查，并且是程序化的而非实例特定的。


Our contributions are as follows:
我们的贡献如下：


- We formulate agent-skill learning as optimization over an external natural-language state and introduce SKILLOPT, a harness-agnostic optimizer with rollout batches, reflection minibatches, add/delete/replace edits, textual learning rates, schedules, held-out acceptance, rejected-edit buffers, and epoch-wise slow/meta update.
- 我们将智能体技能学习表述为在外部自然语言状态上的优化，并提出 SKILLOPT：一种与执行框架无关的优化器，包含 rollout 批次、反思小批次、添加/删除/替换编辑、文本学习率、调度、保留集接受、拒绝编辑缓冲区以及按 epoch 的慢速/元更新。

## 2 Related Work
## 2 相关工作


Prompt auto tuning and agent-configuration search. GEPA demonstrates that trajectory feedback can guide reflective prompt evolution and outperform reinforcement learning on several language-agent tasks [13]. ABSTRAL and EvoTest extend this idea from single prompts to multi-agent design documents and test-time agentic system evolution without gradients or fine-tuning [15, 16]. By treating language artifacts as optimizable objects, these methods can directly exploit execution feedback, but they mainly target prompts, system designs, or full configurations rather than reusable domain adaptation. SKILLOPT instead optimizes a persistent skill document that can be trained, validated, exported, and reused with the adapted model, applying language-level controllability to a stable procedural skill state.
提示自动调优与代理配置搜索。GEPA 表明，轨迹反馈可以引导反思式提示进化，并在若干语言智能体任务上优于强化学习 [13]。ABSTRAL 和 EvoTest 将这一思路从单个提示扩展到多智能体设计文档，以及无梯度、无微调的测试时智能体系统进化 [15, 16]。通过将语言产物视为可优化对象，这些方法可以直接利用执行反馈，但它们主要针对提示、系统设计或完整配置，而非可复用的领域适配。SKILLOPT 则优化一个持久的技能文档，它可被训练、验证、导出，并与适配后的模型一起复用，将语言层面的可控性应用于稳定的程序性技能状态。


Skill construction and skill evolution. SkillsBench and the SoK on agentic skills frame skills as reusable procedural knowledge, covering tool policies, applicability conditions, execution routines, and supporting resources [7, 8]. Prior systems construct such skills from lifelong experience, trajectory lessons, skill knowledge bases, or heterogeneous domain resources [9, 12, 17- 19], and further refine them through failure analysis, creation-evaluation-revision loops, coevolving generators and verifiers, collective updates, or reinforcement learning [10, 11, 20-26]. While these works emphasize skill discovery, repository growth, sharing, evolutionary search, or policy optimization, SKILLOPT studies a narrower problem: how to train one compact domain skill with deep-learning-style controls such as trajectory batches, reflection minibatches, textual learning rates, validation gates, rejected-edit buffers, and slow/meta updates. This yields a controlled and auditable procedure for producing a portable best_skill.md without changing model weights.
技能构建与技能进化。SkillsBench 和关于智能体技能的 SoK 将技能视为可复用的程序性知识，涵盖工具策略、适用条件、执行例程和支撑资源 [7, 8]。先前系统从终身经验、轨迹教训、技能知识库或异构领域资源中构建此类技能 [9, 12, 17- 19]，并通过失败分析、创建-评估-修订循环、协同进化的生成器与验证器、集体更新或强化学习进一步细化 [10, 11, 20-26]。尽管这些工作强调技能发现、仓库扩展、共享、进化搜索或策略优化，SKILLOPT 研究的是一个更窄的问题：如何用类似深度学习的控制方式训练一个紧凑的领域技能，例如轨迹批次、反思小批次、文本学习率、验证门、被拒编辑缓冲区和慢/元更新。这提供了一种受控且可审计的流程，用于生成可移植的 best_skill.md，而无需改变模型权重。

## 3 方法


### 3.1 Problem Setup
### 3.1 问题建模


A skill $s$ is a natural-language policy inserted into the agent context before execution,consistent with recent work treating skills as reusable procedural knowledge for agents [7, 8]. In direct-chat benchmarks, it is prepended to the system or developer instruction; in tool-use harnesses, it becomes persistent procedural memory. We use $M$ to denote the frozen target model whose behavior is being adapted through skill optimization. For a harness $h$ ,task $x$ ,and skill $s$ ,execution produces a trajectory $\tau$ and a scalar score $r$ :
技能 $s$ 是一种自然语言策略，会在执行前被插入到智能体的上下文中，符合将技能视为可复用的过程性知识的相关工作 [7, 8]。在直接对话基准中，它会被置于系统或开发者指令之前；在工具使用框架中，它会变成持久的过程性记忆。我们用 $M$ 表示被冻结的目标模型，其行为正通过技能优化进行适配。对于一个框架 $h$ 、任务 $x$ 和技能 $s$ ，执行会产生轨迹 $\tau$ 与标量得分 $r$ ：


$$
\left( {\tau \left( s\right) ,r\left( s\right) }\right)  = h\left( {M,x,s}\right) ,\;r\left( s\right)  \in  \left\lbrack  {0,1}\right\rbrack  . \tag{1}
$$



Given train,selection,and test splits ${D}_{\mathrm{{tr}}},{D}_{\mathrm{{sel}}},{D}_{\text{ test }}$ ,SKILLOPT uses ${D}_{\mathrm{{tr}}}$ to generate a set of candidate skills $\mathcal{C}\left( {D}_{\mathrm{{tr}}}\right)$ ,selects the best skill on ${D}_{\text{ sel }}$ ,and reports the final performance on ${D}_{\text{ test }}$ :
给定训练、选择与测试划分 ${D}_{\mathrm{{tr}}},{D}_{\mathrm{{sel}}},{D}_{\text{ test }}$ ，SKILLOPT 使用 ${D}_{\mathrm{{tr}}}$ 生成一组候选技能 $\mathcal{C}\left( {D}_{\mathrm{{tr}}}\right)$ ，在 ${D}_{\text{ sel }}$ 上选择最佳技能，并在 ${D}_{\text{ test }}$ 上报告最终性能：


$$
{s}_{\text{ sel }}^{ \star  } = \arg \mathop{\max }\limits_{{s \in  \mathcal{C}\left( {D}_{\mathrm{{tr}}}\right) }}\frac{1}{\left| {D}_{\text{ sel }}\right| }\mathop{\sum }\limits_{{x \in  {D}_{\text{ sel }}}}r\left( s\right) , \tag{2}
$$



$$
\operatorname{Test}\left( {s}_{\text{ sel }}^{ \star  }\right)  = \frac{1}{\left| {D}_{\text{ test }}\right| }\mathop{\sum }\limits_{{x \in  {D}_{\text{ test }}}}r\left( {s}_{\text{ sel }}^{ \star  }\right) . \tag{3}
$$



The training split supplies experience, the selection split gates updates, and the test split is used only for final reporting. The optimizer state contains the current skill, the best validation-gated skill, cached skill hashes, an epoch-local rejected-step buffer, and optional slow/meta-update state. Only the best accepted skill is exported as best_skill.md.
训练划分提供经验，选择划分用于控制更新，测试划分仅用于最终汇报。优化器状态包含当前技能、通过验证门控的最佳技能、已缓存的技能哈希、按轮次本地的被拒步缓冲区，以及可选的慢更新/元更新状态。只有通过接受条件的最佳技能会被导出为 best_skill.md。


### 3.2 Forward Pass: Rollout Evidence
### 3.2 前向传递：滚动证据


At each optimization step,the target model runs a rollout batch from ${D}_{\mathrm{{tr}}}$ with the current skill. The harness records task metadata, messages, tool calls, observations, command outputs, final answers, verifier feedback, and benchmark-specific context such as spreadsheet previews, document references, or compact execution traces. This batch is the evidence unit: small batches update quickly but noisily, while larger batches expose more recurring patterns before the skill changes. The implementation also supports accumulation, where several rollout batches are reflected on separately and merged into one update, decoupling execution throughput from update frequency.
在每个优化步骤中，目标模型都会使用当前技能从 ${D}_{\mathrm{{tr}}}$ 运行一个滚动批次。执行器会记录任务元数据、消息、工具调用、观测、命令输出、最终答案、验证器反馈，以及特定基准的上下文，例如电子表格预览、文档引用或紧凑的执行轨迹。这个批次就是证据单元：小批次更新更快但更嘈杂，而大批次在技能变化之前能暴露更多重复出现的模式。实现还支持累积，即将多个滚动批次分别反映后合并为一次更新，从而将执行吞吐量与更新频率解耦。


### 3.3 Backward Pass: Minibatch Reflection
### 3.3 反向传递：小批量反思


The optimizer model turns trajectories into skill edits, following the broader line of trajectory-driven reflection and prompt evolution [13, 27, 28]. It first separates failures from successes and partitions each group into reflection minibatches. This matters because single trajectories often produce anecdotal fixes, while minibatches expose reusable procedural errors: the agent consistently searches the wrong source, writes an answer in the wrong format, or fails to verify a tool result. Failure minibatches propose missing or corrective rules; success minibatches preserve behaviors that already work. Each reflection returns structured add/delete/replace edits, or in rewrite mode a small set of rewrite suggestions.
优化器模型将轨迹转化为技能编辑，延续轨迹驱动的反思与提示演化这一更宏观思路[13, 27, 28]。它首先将失败与成功分开，并将每一组划分为反思小批次。这一点很关键：单条轨迹往往只能带来轶事式的修补，而小批次能够暴露可复用的程序性错误——智能体会持续从错误来源检索、以错误格式给出答案，或未能核验工具结果。失败小批次会提出缺失的或纠正性的规则；成功小批次则保留那些已经有效的行为。每次反思都会返回结构化的添加/删除/替换编辑；在改写模式下，则返回一小组改写建议。


Local proposals are merged hierarchically by first consolidating failure- and success-driven edits separately, then combining them with priority on failure corrections. This step filters duplicate, contradictory, and example-specific suggestions before the optimizer selects the final bounded update.
局部提案会通过先分别合并由失败与成功驱动的编辑、再在优先处理失败修正的前提下将两者合并的方式进行分层整合。在优化器选择最终的有界更新之前，这一步会过滤重复的、互相矛盾的以及过于依赖示例的建议。


### 3.4 Bounded Text Updates
### 3.4 受限文本更新


The learning-rate analogue in SKILLOPT is the edit budget ${L}_{t}$ : the maximum number of skill edits applied at step $t$ . After aggregation,the optimizer model ranks the merged edit pool by expected utility and clips it to the top ${L}_{t}$ edits. This is the key difference from ad hoc prompt rewriting. Unbounded rewrites can erase useful rules, introduce incompatible instructions, or overfit to a local failure; bounded updates preserve continuity while still allowing the skill to acquire new procedures. SKILLOPT supports constant, linear, cosine, and autonomous schedules. The default cosine schedule starts with larger edits and decays toward smaller consolidation steps.
在 SKILLOPT 中，学习率对应项是编辑预算 ${L}_{t}$：即在步骤 $t$ 时应用的技能编辑的最大数量。聚合之后，优化器模型会按期望效用对合并的编辑池进行排序，并将其裁剪为前 ${L}_{t}$ 条编辑。这一点是它相较于临时性的提示重写的关键差异。无限制的重写可能会抹除有用规则、引入不兼容的指令，或对局部失败过拟合；受限更新在保持连贯性的同时，仍允许技能习得新的操作流程。SKILLOPT 支持恒定、线性、余弦以及自主式的调度。默认的余弦调度从更大的编辑开始，并衰减到更小的整合步骤。


The selected edits produce a candidate skill. In patch mode, edits are localized operations such as append, insert, replace, and delete; in rewrite mode, selected suggestions condition a full skill rewrite. Step-level edits cannot overwrite the protected slow-update field, so fast local changes and slower epoch-wise consolidation remain separated.
所选编辑将生成一个候选技能。在补丁模式下，编辑是诸如追加、插入、替换与删除之类的局部操作；在重写模式下，所选建议将用于条件化一次完整的技能重写。步骤级编辑无法覆盖受保护的慢更新字段，因此快速的局部改动与较慢的按轮次整合仍保持分离。


### 3.5 Validation Gate and Rejected-Edit Buffer
### 3.5 验证门与拒绝编辑缓冲区


Every candidate skill is evaluated on ${D}_{\text{ sel }}$ with the same frozen target model and harness. If it improves over the current selection score, it becomes the new current skill; if it also exceeds the best score so far, it becomes best_skill.md. Otherwise it is rejected. This gate turns reflection into propose-and-test optimization rather than unconditional self-editing, which is crucial because plausible textual diagnoses can still hurt the actual target model.
每个候选技能都会在${D}_{\text{ sel }}$上使用同一个冻结的目标模型和工具链进行评估。如果它优于当前选择分数，就成为新的当前技能；如果它也超过迄今为止的最佳分数，就成为best_skill.md。否则它会被拒绝。这个门控将反思转化为提议与测试式优化，而不是无条件自我编辑，这一点至关重要，因为看似合理的文本诊断仍可能损害实际的目标模型。


Rejected updates are still useful. The optimizer records an epoch-local buffer containing observed failure patterns and, for rejected steps, the edits that were tried and the score drop they caused. Later reflection calls in the same epoch receive this buffer, so the optimizer model can avoid repeating failed edits and focus on unresolved failures. This gives the loop negative feedback during training without adding inference-time cost.
被拒绝的更新仍然有用。优化器会记录一个仅在当前 epoch 内有效的缓冲区，其中包含观察到的失败模式，以及对于被拒绝步骤，曾尝试过的编辑及其导致的分数下降。后续在同一 epoch 内的反思调用会接收这个缓冲区，因此优化器模型可以避免重复失败的编辑，并专注于尚未解决的失败。这使得循环在训练期间获得负反馈，而不会增加推理时成本。


### 3.6 Epoch-Wise Slow/Meta Update
### 3.6 基于轮次的慢更新/元更新


Fast updates learn from the current batch; the epoch-wise slow/meta update learns from adjacent epochs. At the end of an epoch, SKILLOPT samples the same training items under the previous epoch's skill and the current skill, then groups them into improvements, regressions, persistent failures, and stable successes. The optimizer model writes a concise longitudinal guidance block into a protected slow-update field, and this candidate is still passed through the validation gate. Thus slow update captures durable domain lessons while preserving the same safety check as step-level edits.
快速更新从当前批次中学习；基于轮次的慢更新/元更新则从相邻轮次中学习。在每个轮次结束时，SKILLOPT 会在上一轮次的技能与当前技能下对相同的训练样本进行抽样，然后将其分组为改进、退步、持续失败与稳定成功。优化器模型会将一段简洁的纵向指导块写入受保护的慢更新字段，该候选项仍会经过验证门。因此，慢更新在保留与步骤级编辑相同安全检查的同时，能够捕捉持久的领域经验。


The meta skill is optimizer-side only. It summarizes which edit patterns helped, which were rejected, and which failures persisted across epochs. This meta guidance is prepended to future optimizer prompts for reflection, merging, and ranking, but it is not shipped with the target model. The advantage is separation of concerns: the deployed skill remains compact and portable, while training benefits from a richer record of the editing process.
元技能仅在优化器侧。它会概括哪些编辑模式带来了帮助、哪些被拒绝、以及哪些失败在各轮次间持续存在。这一元指导会被预置到未来优化器提示中，用于反思、融合与排序，但不会随目标模型一起交付。优势在于职责分离：部署后的技能保持精炼且便于携带，而训练则能从更丰富的编辑过程记录中获益。


### 3.7 Harness-Agnostic Deployment
### 3.7 与执行框架无关的部署


SKILLOPT is harness-agnostic through a lightweight adapter interface, matching the broader trend toward agents embedded in tool-use and software-execution environments [1, 2, 4]. An adapter constructs train/evaluation batches, injects the current skill into the agent context, runs the native harness, and returns scored trajectories. The same optimizer therefore works for direct QA, spreadsheet execution, document reasoning, multimodal QA, embodied environments, and Codex-style or Claude Code-style execution loops. This is the main practical advantage of treating skills as the adaptation layer: a stronger optimizer model can train a reusable skill artifact offline, and the resulting best_skill.md can then be deployed or tested across target models, harnesses, and nearby benchmarks without changing model weights.
SKILLOPT 通过一个轻量级适配器接口实现与执行框架无关，这也契合了代理嵌入工具使用和软件执行环境的更大趋势 [1, 2, 4]。适配器负责构建训练/评估批次，将当前技能注入代理上下文，运行原生执行框架，并返回带评分的轨迹。因此，同一个优化器可用于直接问答、表格执行、文档推理、多模态问答、具身环境，以及 Codex 风格或 Claude Code 风格的执行循环。将技能视为适配层的主要实际优势在于：更强的优化器模型可以离线训练出可复用的技能工件，而生成的 best_skill.md 随后可在目标模型、执行框架和邻近基准上部署或测试，而无需更改模型权重。


## 4 Experiments
## 4 实验


We evaluate SKILLOPT as a text-space optimizer for frozen agents: the target model executes each task with the current skill, while an offline optimizer edits that skill from rollout evidence. The experiments answer four questions. (i) Do optimized skills improve over no-skill, human-skill, one-shot LLM-skill, prompt-optimization (TextGrad, GEPA), and skill-evolution (Trace2Skill, EvoSkill) baselines? (ii) Does the same loop work across direct chat, Codex, and Claude Code harnesses, and across seven target models from frontier-scale GPT to small Qwen? (iii) Which optimizer controls matter? (iv) What do the learned skills look like, and at what cost?
我们将 SKILLOPT 评估为针对冻结智能体的文本空间优化器：目标模型在当前技能下执行每个任务，而离线优化器会基于回放证据来编辑该技能。实验回答四个问题。（i）优化后的技能是否优于无技能、人类技能、一次性 LLM 技能、提示优化（TextGrad、GEPA）以及技能演化（Trace2Skill、EvoSkill）等基线？（ii）同一套循环能否在直接对话、Codex 以及 Claude Code 的工具框架中工作，并能否覆盖从前沿规模的 GPT 到小型 Qwen 的七个目标模型？（iii）哪些优化器控制项至关重要？（iv）学到的技能是什么样的，代价又是什么？


Setting. We report each benchmark's native hard score or exact-match accuracy on held-out test splits across SearchQA [29], SpreadsheetBench [30], OfficeQA [31], DocVQA [32], LiveMathemati-cianBench [33] (abbreviated LiveMath in tables), and ALFWorld [34], using two model families: GPT [35] and Qwen [36, 37]. The benchmark suite is intentionally diverse-it spans single-round QA (SearchQA, DocVQA, LiveMathematicianBench MCQ), multi-turn tool loops with up to 24 tool calls (OfficeQA), multi-round codegen with up to 30 turns and a real openpyxl/pandas runtime (SpreadsheetBench, default mode=multi), and persistent embodied interaction with up to 50 steps per episode (ALFWorld). Dataset-backed runs use deterministic train/selection/test splits derived from the same dataset seed (split_seed = 42); the selection split is used only to accept or reject candidate skill edits, and all reported scores are computed on the disjoint held-out test split. The reported numbers thus measure generalization, not validation-set fit.
设置。我们在 SearchQA [29]、SpreadsheetBench [30]、OfficeQA [31]、DocVQA [32]、LiveMathemati-cianBench [33]（表中缩写为 LiveMath）和 ALFWorld [34] 的保留测试划分上，报告各基准的原生硬分数或精确匹配准确率；使用两类模型家族：GPT [35] 与 Qwen [36, 37]。基准套件刻意保持多样性——它涵盖单轮问答（SearchQA、DocVQA、LiveMathematicianBench MCQ）、最多 24 次工具调用的多轮工具循环（OfficeQA）、最多 30 轮的多轮代码生成，并在真实 openpyxl/pandas 运行环境下执行（SpreadsheetBench，默认模式=multi），以及每个 episode 最多 50 步的持续具身交互（ALFWorld）。基于数据集的运行使用由同一数据集种子生成的确定性训练/选择/测试划分（split_seed = 42）；选择划分仅用于决定接受或拒绝候选技能编辑，而所有报告分数都在互斥的保留测试划分上计算。因此，所报告的数值衡量的是泛化能力，而非验证集拟合。

Harnesses. Direct chat invokes the target model through a single chat completion call with the skill prepended to the system prompt. The Codex harness drives the target through the codex CLI in a workspace-write sandbox [38]; SKILLOPT renders the current skill to a per-task SKILL.md alongside task files and reads back a compact execution trace (codex_trace_summary.txt) that is included in the teacher reflection context, so the optimizer learns from what the agent actually did, not just its final answer. The Claude Code harness mirrors the same workspace contract through the claude CLI [39]. All three modes consume the same best_skill.md file format, which is what enables the cross-harness transfer experiments in Section 4.3.
工具箱。Direct 通过一次聊天完成调用来触发目标模型：将技能前置到 system 提示中。Codex 工具箱通过 codex CLI 在可写工作区沙盒中驱动目标[38]；SKILLOPT 将当前技能渲染为每个任务的 SKILL.md，放在任务文件旁边，并读取一段紧凑的执行轨迹（codex_trace_summary.txt），将其包含进教师反思上下文中；因此优化器学习的是智能体实际做了什么，而不仅仅是其最终答案。Claude Code 工具箱同样通过 claude CLI [39] 复刻相同的工作区契约。三种模式都消耗同一种最佳技能 best_skill.md 文件格式，这也是第4.3节开展跨工具箱迁移实验的前提。


Baselines. We compare against seven baselines that span the no-adaptation, hand-written, one-shot, and learning families: no skill (frozen target model run with the benchmark's default system prompt), human skill (an expert-written skill document curated per benchmark), one-shot LLM skill (a single skill generated from a high-level task description by GPT-5.5 and never updated), Trace2Skill [9] (trajectory-level skill distillation), TextGrad [40] (gradient-style natural-language prompt optimization), GEPA [13] (Pareto reflective prompt evolution), and the harness-side competitor EvoSkill [10] (skill-folder evolution under failure analysis). All baselines use the same target model, the same held-out test split, and the same scorer for every benchmark, so the comparison isolates the choice of adaptation procedure rather than secondary factors such as prompt template or scoring pipeline.
基线。我们与七类基线对比，覆盖不适配、人工编写、单次提示以及学习型方法：无技能（冻结的目标模型运行，使用基准的默认 system 提示）、人类技能（每个基准由专家撰写并策划的技能文档）、单次 LLM 技能（由 GPT-5.5 根据高层任务描述生成的单一技能，且从不更新）、Trace2Skill [9]（基于轨迹层面的技能蒸馏）、TextGrad [40]（梯度式自然语言提示优化）、GEPA [13]（帕累托式反思提示演化），以及工具箱侧的对手 EvoSkill [10]（在失败分析下进行技能文件夹演化）。所有基线都使用相同的目标模型、相同的保留测试划分，以及每个基准相同的评分器，因此比较能够将差异归因于适配流程的选择，而非诸如提示模板或评分流水线这类次要因素。

Taken together, the table supports a strong empirical claim: across direct chat and two tool-execution harnesses, across seven target models, and on procedural and factual benchmarks alike, optimizing a single compact skill artifact under bounded text-space training is the strongest no-weight-update adaptation strategy among the baselines we consider. The main gains come from feedback-driven skill editing rather than from a better one-shot prompt: human and LLM skills can help when prior instructions happen to match the benchmark, but they cannot correct failures after observing rollouts; Trace2Skill mines trajectory lessons without a held-out gate; TextGrad and GEPA optimize prompts but not a persistent skill artifact; and EvoSkill, the strongest harness-side competitor, lacks both bounded textual learning rates and rejected-edit memory. These comparisons support the central design choice-keep the target model, harness, and evaluator fixed, and optimize only the reusable skill artifact.
综上，该表有力支持一个经验性结论：在直接对话和两个工具执行评测基准中，在七个目标模型上，以及在程序性与事实性基准上，针对受限文本空间训练优化单一紧凑技能工件，是我们所考虑基线中最强的无权重更新适配策略。主要收益来自基于反馈的技能编辑，而非更好的“一次性提示”：当先前指令恰好与基准匹配时，人类与LLM技能能有所帮助，但它们无法在观察到回合结果后纠正失败；Trace2Skill无需外部的隔离门也能挖掘轨迹经验；TextGrad和GEPA优化的是提示而非持续的技能工件；而EvoSkill（最强的基准端竞争者）既没有受限的文本学习率，也缺少被拒绝编辑的记忆。这些对比支持核心设计选择：固定目标模型、评测工具与评估器，仅优化可复用的技能工件。

Gate strictness and edit observability. The validation gate is intentionally strict: a candidate skill is accepted only when its selection-split score is strictly greater than the current selection score, so ties are rejected and the deployed skill never silently drifts. This conservative criterion makes rejected edits informative negative feedback rather than hidden state. Operationally, every step also records an edit_apply_report.json containing per-edit accept/skip status, so the source of every change to best_skill.md is recoverable after the fact. The epoch-wise slow/meta update writes into a markup-fenced protected region of the skill document that step-level edits cannot overwrite, separating the fast intra-epoch update from the slower cross-epoch consolidation; the optimizer-side meta skill lives only in the teacher's reflection context and is never shipped with the deployed artifact. These implementation choices explain why removing both meta skill and slow update is especially damaging on SpreadsheetBench: it removes the long-horizon evidence stream and the protected-region contract that keeps local edits from overwriting durable procedural lessons.
门控严格性与编辑可观测性。验证门控刻意设得很严格：只有当候选技能的 selection-split 分数严格高于当前 selection 分数时才会被接受，因此平局会被拒绝，已部署的技能也绝不会无声漂移。这一保守准则使被拒绝的编辑成为有信息量的负反馈，而非隐藏状态。操作上，每一步还会记录一个 edit_apply_report.json，其中包含每次编辑的接受/跳过状态，因此事后可追溯 best_skill.md 每次变更的来源。按 epoch 的慢速/meta 更新会写入技能文档中一个带标记围栏保护的区域，步骤级编辑无法覆盖，从而将 epoch 内的快速更新与跨 epoch 的较慢整合分离开来；优化器侧的 meta skill 只存在于教师的反思上下文中，绝不会随部署产物一同发布。这些实现选择解释了为什么同时移除 meta skill 和 slow update 在 SpreadsheetBench 上尤其有害：它去除了长期证据流，以及防止局部编辑覆盖持久过程性经验的受保护区域契约。


Overall, the ablations show that the gains are relatively insensitive to the exact rollout batch, reflection minibatch, or learning-rate schedule, but much more sensitive to the presence of bounded text-space learning, validation gating, rejected-edit feedback, and epoch-wise slow/meta update-the design choices that make skill editing behave like a controlled training loop.
总体而言，消融结果表明，收益对精确的 rollout batch、reflection minibatch 或学习率调度相对不敏感，但对有界文本空间学习、验证门控、被拒绝编辑反馈以及按 epoch 的 slow/meta update 的存在要敏感得多——这些设计使技能编辑表现得像一个受控的训练循环。

### 4.3 分析与迁移


Tables 4-4 ask whether an optimized skill behaves like a reusable artifact rather than a task-specific prompt. We test three shifts: deploying a skill across model scales (Table 4), moving it across execution harnesses (Table 4), and applying it to nearby math benchmarks, including OlympiadBench [41] and Omni-MATH [14] (Table 4). Table 5 then asks how much of the gain depends on optimizer capacity by replacing the frontier optimizer with a target-matched one of the same scale as the deployed model.
表 4-4 询问：被优化的技能，是否更像可复用的产物，而不是特定任务的提示词。我们测试三种变化：将技能跨模型规模部署（表 4）、将其跨执行工具框架迁移（表 4），以及将其应用到相邻的数学基准，包括奥林匹克竞赛基准 OlympiadBench [41] 和 Omni-MATH [14]（表 4）。随后表 5 再问：收益有多少取决于优化器能力——通过把前沿优化器替换为一个目标匹配的优化器（规模与部署模型相同）。


Cross-model transfer. Table 4(a) reports four genuine cross-model transfers (the GPT-5.4 $\rightarrow$ GPT- 5.4 entries are kept only as in-domain references and marked - in the transferred column, since source and target match): SpreadsheetBench skills trained on GPT-5.4 transfer to GPT-5.4-mini (+9.4) and GPT-5.4-nano (+3.0), and LiveMath skills transfer to GPT-5.4-mini (+4.5) and GPT- 5.4-nano (+5.6). All four cross-model rows are positive, and on one of them the transferred skill surpasses the in-domain SKILLOPT reference (LiveMath GPT-5.4-nano: 28.8 transferred vs. 27.2 in-domain), suggesting that some learned procedures are target-model agnostic. The remaining cross-model rows still recover a useful fraction of the in-domain gain -e.g. SpreadsheetBench GPT-5.4-mini retains 82% of the in-domain gain (+9.4 of +11.4) and no row falls below the target's no-skill baseline.
跨模型迁移。表 4(a) 报告了四条真正的跨模型迁移（GPT-5.4 $\rightarrow$ 的 GPT- 5.4 条目仅作为域内参照保留，并在迁移列中标注——由于源模型与目标模型匹配）：在 GPT-5.4 上训练的 SpreadsheetBench 技能迁移到 GPT-5.4-mini（+9.4）和 GPT-5.4-nano（+3.0）；LiveMath 技能则迁移到 GPT-5.4-mini（+4.5）和 GPT- 5.4-nano（+5.6）。四条跨模型结果均为正，其中一条迁移后的技能超过了域内的 SKILLOPT 参照（LiveMath GPT-5.4-nano：迁移 28.8 对比域内 27.2），这表明部分学到的流程与目标模型无关。其余跨模型行仍能恢复域内收益的一部分——例如 SpreadsheetBench 在 GPT-5.4-mini 上保留了 82% 的域内收益（+9.4 / +11.4），且没有任何一条低于目标的无技能基线。


Cross-harness transfer. The harness-shift rows in Table 4(b) are the clearest deployment signal. A SpreadsheetBench skill trained inside the Codex loop transfers to Claude Code with absolute gain +59.7 over the Claude Code no-skill baseline (22.1 $\rightarrow  {81.8}$ ,slightly exceeding the in-domain Claude Code SKILLOPT reference of 80.4), and the symmetric Claude-Code $\rightarrow$ Codex transfer adds +43.6 on top of the Codex baseline (27.5→71.1). On LiveMath, the Codex→Claude Code transfer is smaller (+1.6 over a 40.8 baseline) but still positive, while the Claude-Code $\rightarrow$ Codex transfer adds +12.8 (35.2 $\rightarrow  {48.0}$ ). Because the two harnesses expose different tool/file APIs and command surfaces, these positive transfers suggest that the learned rules are not only harness-specific command recipes. In SpreadsheetBench especially, the transferred skill appears to encode workbook-level procedures such as structure-first inspection, formula-aware verification, and static-value materialization, so the cost of optimizing a skill in one execution environment can be amortized across related deployment environments.
跨工具框架迁移。表 4(b) 中“框架变更”相关的行，是最清晰的部署信号。在 Codex 循环内部训练的 SpreadsheetBench 技能，迁移到 Claude Code 后相对 Claude Code 无技能基线带来绝对收益 +59.7（22.1 $\rightarrow  {81.8}$ ，略高于域内 Claude Code 的 SKILLOPT 参照 80.4）；对称的 Claude-Code $\rightarrow$ Codex 迁移在 Codex 基线之上再加 +43.6（27.5→71.1）。在 LiveMath 上，Codex→Claude Code 的迁移更小（相对 40.8 的基线为 +1.6），但仍为正；而 Claude-Code $\rightarrow$ Codex 迁移再加 +12.8（35.2 $\rightarrow  {48.0}$ ）。由于这两个框架暴露了不同的工具/文件 API 与命令界面，这些正向迁移表明，学到的规则不仅仅是框架特定的指令配方。尤其在 SpreadsheetBench 中，迁移后的技能似乎编码了工作簿级流程，如先结构后检查、公式感知的验证，以及静态值的物化；因此，在一个执行环境中优化技能的成本，可以在相关的部署环境中摊销。


Cross-benchmark transfer. Cross-benchmark transfer is the strictest of the three shifts: source and target benchmarks share only the broad task family (math). On the OlympiadBench $\rightarrow$ Omni-MATH direction reported in Table 4(c), the transferred skill is positive on all three model scales we evaluate, with gains of +3.7 on GPT-5.4, +1.8 on GPT-5.4-mini, and +1.3 on GPT-5.4-nano. These rows are smaller than the in-domain and cross-harness transfers unsurprisingly, since they require the optimized skill to retain useful procedural knowledge after both the test instances and the answer-format conventions change - but they remain uniformly positive, supporting the intended interpretation that the optimized skill encodes reusable mathematical procedure rather than memorized benchmark-specific formatting.
跨基准迁移。跨基准迁移是三种变化中最严格的一种：源基准与目标基准仅共享广义任务家族（数学）。在表 4(c) 报告的 OlympiadBench $\rightarrow$ 向 Omni-MATH 方向上，被迁移的技能在我们评估的三种模型规模上都为正，增益分别为：GPT-5.4 +3.7、GPT-5.4-mini +1.8、GPT-5.4-nano +1.3。正如预期，这些行小于域内迁移与跨工具框架迁移，因为它要求在测试实例与答案格式约定都发生变化后，仍保留有用的程序性知识——但结果依然一致为正。它支持我们期望的解释：被优化的技能编码的是可复用的数学程序，而不是记忆了特定基准的格式化内容。


Effect of optimizer strength. Because the optimizer in SKILLOPT runs only during the offline training loop and is never invoked at deployment, optimizer choice is a training-time lever: a stronger optimizer can improve the deployed skill without raising the inference cost of using that skill. The deployed artifact is still a static best_skill.md that calls only the target model. Table 5 quantifies this lever by running the same loop with two optimizer regimes—a strong frontier optimizer (GPT-5.5) and a target-matched optimizer that shares the target model-while holding the rollout batches, validation gate, bounded edit budget, rejected-edit buffer, and slow/meta update identical.
优化器强度的影响。由于 SKILLOPT 中的优化器只在离线训练循环期间运行，从不在部署时调用，因此优化器选择是一个训练时杠杆：更强的优化器可以提升部署后的技能，而不会增加使用该技能的推理成本。部署后的产物仍然是一个静态的 best_skill.md，只会调用目标模型。表 5 通过在两种优化器策略下运行相同的循环量化了这一杠杆——一种是强前沿优化器（GPT-5.5），另一种是目标匹配的优化器：同时保持 rollout 批次、验证门控、受限编辑预算、被拒绝编辑缓冲区以及慢更新/元更新设置完全相同，仅替换优化器。


Two observations follow. First, the stronger optimizer produces larger absolute gains on every (benchmark, target) cell we test: GPT-5.4-nano lifts by +19.0 vs. +11.9 on SpreadsheetBench and +19.0 vs. +14.1 on SearchQA, and GPT-5.4-mini follows the same ordering (+11.4 vs. +7.1 on SpreadsheetBench, +4.3 vs. +2.4 on SearchQA). The bounded-edit, validation-gated loop is what makes this monotone: without the gate, a stronger optimizer could just as easily push larger but harmful rewrites. Second, the target-matched optimizer is far from collapsed-it recovers 56-74% of the strong-optimizer gain across the four cells, confirming that SKILLOPT is not a distillation pipeline from a stronger teacher into a weaker student: the optimization loop itself contributes substantial value on top of whatever the optimizer can already do. The practical implication is that a high-capacity frontier optimizer is the right default whenever it is available-it costs only training-time API calls and adds nothing to deployment-while the same loop remains effective if the budget forces a target-matched optimizer instead.
有两个观察。首先，更强的优化器在我们测试的每个（基准，目标）单元上都带来更大的绝对增益：GPT-5.4-nano 在 SpreadsheetBench 上提升 +19.0，对比 +11.9，在 SearchQA 上提升 +19.0，对比 +14.1；GPT-5.4-mini 也遵循同样的排序（SpreadsheetBench 上 +11.4 对比 +7.1，SearchQA 上 +4.3 对比 +2.4）。这种单调性正是由受限编辑、验证门控的循环所带来的：如果没有这个门控，更强的优化器完全可能进行更大但有害的改写。其次，目标匹配的优化器远未失效——它在四个单元中恢复了强优化器增益的 56-74%，这也证实 SKILLOPT 不是把更强教师蒸馏成更弱学生的流水线：优化循环本身就在优化器原本能做的基础上提供了可观价值。实际意义在于，只要高容量的前沿优化器可用，它就是默认的最佳选择——它只会增加训练阶段的 API 调用，而不会增加部署成本；而当预算迫使使用目标匹配的优化器时，这一循环同样依然有效。

### 4.4 Learned Skills: Compactness, Cost, and Examples
### 4.4 学到的技能：紧凑性、成本与示例


A central premise of SKILLOPT is that the trainable object should remain a small, inspectable text document. Tables 1-5 demonstrate that the optimizer is effective; this subsection asks what its output actually looks like and what it costs. We characterize the learned artifact on three axes-compactness, edit economy, and cost-per-point-and then show one representative learned rule per benchmark to illustrate what kind of procedural knowledge survives the bounded-update loop.
SKILLOPT 的核心前提是：可训练对象应保持为一份小且可检查的文本文档。表 1-5 证明优化器是有效的；本小节将回答：它的输出究竟是什么样的，以及它要付出什么代价。我们从三个维度表征所学产物——紧凑性、编辑经济性以及单位点的成本——然后在每个基准上各展示一个代表性的学到规则，以说明在有界更新循环中，哪种类型的程序性知识仍会存活。


Compactness. The final skills are uniformly small. Across the six benchmarks in Table 6, the final best_skill.md ranges from 379 tokens (LiveMathematicianBench) to 1,995 tokens (SpreadsheetBench), with a median of roughly 920 tokens. Even the longest learned skill is well below a typical system-prompt budget for modern frontier models, and the shortest one fits inside a single screen. The growth from initial to final skill is moderate $( \times  {2.5}$ to $\times  {53}$ depending on whether the initial skill was a one-liner or a paragraph), but the final size in absolute tokens stays small enough that a domain practitioner can read, audit, and edit the deployed artifact in minutes.

Benchmarks with longer trajectories or richer multimodal context—SearchQA (37.9M / pt) and DocVQA (46.4M / pt) - cost an order of magnitude more per point. The important deployment distinction is that this cost is paid once during skill training; after export, the optimized best_skill.md adds no optimizer calls, no weight updates, and only a compact text artifact to the target agent.
每个测试集增益点的成本。训练token列量化了运行该循环的代价。可以看到两种情形。程序性基准——运行很短且很便宜的 SpreadsheetBench、OfficeQA、LiveMathematicianBench——每个绝对测试集点达到0.6-3.6M训练token，即便这些基准上的绝对增益是最大的（例如：在OfficeQA上+39.0分，消耗1.1M tokens/点，总计20.8M tokens）。而在轨迹更长或多模态上下文更丰富的基准上——SearchQA（37.9M/pt）和DocVQA（46.4M/pt）——每点成本高出一个数量级。重要的部署差异在于：这笔成本只在技能训练阶段支付；导出之后，优化得到的best_skill.md不会再调用任何优化器、不会更新权重，只向目标智能体添加一个紧凑的文本产物。


What does a learned skill actually say? Figure 4 reproduces one representative learned rule per benchmark, taken verbatim from the final best_skill.md of each case study in Table 6. Three observations stand out. First, the rules are procedural rather than instance-specific: none of them name a specific question, file, or entity. Second, they consistently encode the discipline that frontier models lack zero-shot: answer-format constraints (OfficeQA, LiveMathematicianBench), evidence binding to a specific visual region (DocVQA), workbook-structure-first reasoning (Spread-sheetBench), search-frontier discipline (ALFWorld), and canonical-entity choice (SearchQA). Third, they read like rules a thoughtful human practitioner would write after a day with the benchmark-except they are produced automatically by the optimizer and validated edit-by-edit on held-out data.
已学技能究竟说了什么？图4针对每个基准复现了一条代表性的已学规则，逐字取自表6中各案例研究的最终 best_skill.md。可以突出三点观察。第一，这些规则是程序性的而非针对具体实例：它们都没有点名某个特定问题、文件或实体。第二，它们持续编码前沿模型在零样本下缺乏的纪律：答案格式约束（OfficeQA、LiveMathematicianBench）、绑定到特定视觉区域的证据（DocVQA）、先以工作簿结构为核心的推理（Spread-sheetBench）、搜索前沿的纪律（ALFWorld）以及规范实体的选择（SearchQA）。第三，它们读起来像是一个深思熟虑的人类从业者在与该基准共度一天之后会写下的规则——不同之处在于：这些规则是由优化器自动生成的，并且在持出数据上逐次编辑验证。


Implications. Together, the four observations above support a stronger version of the central claim. Compactness (<2,000 tokens) and edit economy (1-4 accepted edits) mean the deployed artifact is interpretable. Cost-per-point (0.6M-46.4M tokens / point) shows that the training cost is measurable and paid before deployment. The shape of the learned rules—procedural, generalizable, and consistent with what a thoughtful human practitioner would write - is evidence that text-space optimization with bounded updates and validation gating discovers transferable procedural knowledge rather than merely overfitting to the training split. This complements the cross-model, cross-harness, and cross-benchmark transfer evidence in Section 4.3: the artifact transfers because many of the rules it encodes are intrinsically transferable.
含义。综合上述四点观察，更强版本的核心论断得到支持。紧凑性（<2,000 tokens）和编辑经济性（1-4次被接受的编辑）意味着已部署的产物具有可解释性。按点计的成本（0.6M-46.4M tokens/点）表明训练成本是可度量的，并且在部署之前就已支付。已学规则的形态——程序性、可泛化性，并且与一个深思熟虑的人类从业者会写下的内容一致——证据显示：在受限更新与验证门控的条件下进行的文本空间优化，发现的是可迁移的程序性知识，而不只是对训练分割的过拟合。这与第4.3节中的跨模型、跨工具和跨基准迁移证据相互补充：因为它编码的许多规则本身就具有可迁移性，故该产物能够迁移。


### 4.5 Qualitative Skill Evolution
### 4.5 定性技能演化


We inspect two representative runs to understand what the optimized skill actually learns. The ALFWorld case uses GPT-5.4-nano as the student and GPT-5.5 as the teacher, while the SpreadsheetBench case uses GPT-5.5 as both the frozen student and optimizer model. In both cases, SKILLOPT does not replace the initial skill with an unrelated prompt. Instead, accepted edits add compact procedural constraints around recurring failure modes observed in rollout trajectories.
我们检查两个代表性运行，以理解优化后的技能实际学到了什么。ALFWorld 案例使用 GPT-5.4-nano 作为学生、GPT-5.5 作为教师，而 SpreadsheetBench 案例则将 GPT-5.5 同时用作冻结学生和优化器模型。在这两种情况下，SKILLOPT 并不是用一个无关的提示来替换初始技能。相反，被接受的编辑会围绕回滚轨迹中观察到的重复失败模式，增加简洁的过程性约束。


ALFWorld. The initial ALFWorld skill gives a generic household plan: search for the target object, pick it up, transform it if needed, and place it at the destination. The accepted edits make this plan more stateful and less loop-prone. The optimized skill learns exact object-name matching, so related objects such as mugs, cups, pans, and pots are not substituted for one another. It adds visited-location memory, so unvisited receptacles and surfaces are preferred over repeatedly checking likely but exhausted locations. It also adds destination memory, pick-two progress locks, and direct completion rules: once the agent can clean, heat, cool, place, or otherwise complete the next subgoal, it should take that admissible action instead of examining, closing, or verifying again. Qualitatively, the skill evolves from a general search-transform-place strategy into a finite-state execution policy with object identity, search memory, progress locks, and loop breakers. In this representative run, the selected skill improves ALFWorld held-out test performance from 49.3 to 74.6.
ALFWorld。初始的 ALFWorld 技能给出的是一个通用的家务计划：搜索目标物体、将其拾起、在需要时进行变换，并把它放到目的地。被接受的编辑让这个计划更具状态性，也更不容易陷入循环。优化后的技能学会了精确的物体名称匹配，因此诸如 mug、cup、pan 和 pot 之类的相关物体不会彼此替代。它增加了已访问位置记忆，因此会优先查看未访问的容器和表面，而不是反复检查那些看似可能但已经耗尽的地点。它还增加了目的地记忆、双步进度锁以及直接完成规则：一旦智能体可以清洁、加热、冷却、放置，或以其他方式完成下一个子目标，就应采取该可接受动作，而不是再次检查、关闭或验证。定性地说，这个技能从通用的搜索-变换-放置策略演化为一种有限状态执行策略，包含物体身份、搜索记忆、进度锁和循环终止器。在这一代表性运行中，所选技能将 ALFWorld 留出测试性能从 49.3 提升到 74.6。


SpreadsheetBench. The initial SpreadsheetBench skill already instructs the agent to use Python spreadsheet libraries and preserve unrelated workbook content. The accepted edits turn this generic automation workflow into a workbook-forensics policy. The optimized skill learns to inspect the actual workbook rather than rely on previews, locate headers and target ranges across multiple sheets, normalize keys and cell types before lookup or aggregation, and preserve formatting during structural edits. It also adds a key rule for formula-style prompts: when the grader reads cell values, the agent should compute and write evaluated static values, even if the prompt mentions formulas such as INDEX/MATCH or XLOOKUP. Later edits further require filling complete target ranges, including currently blank result cells, keeping helper computations in Python rather than adding workbook artifacts, and reopening the saved workbook to check boundary rows and remaining blanks. In this representative run, the selected skill improves SpreadsheetBench held-out test performance from 40.4 to 78.9.
SpreadsheetBench。初始的 SpreadsheetBench 技能已经指示智能体使用 Python 电子表格库并保留无关的工作簿内容。被接受的编辑把这种通用自动化工作流变成了一种工作簿取证策略。优化后的技能学会了检查实际工作簿，而不是依赖预览，跨多个工作表定位表头和目标范围，在查找或聚合前规范化键和单元格类型，并在结构性编辑过程中保留格式。它还为公式式提示增加了一条关键规则：当评审器读取单元格值时，智能体应计算并写入求值后的静态值，即使提示中提到诸如 INDEX/MATCH 或 XLOOKUP 之类的公式。后续编辑进一步要求填满完整的目标范围，包括当前为空的结果单元格，将辅助计算保留在 Python 中而不是添加工作簿工件，并重新打开已保存的工作簿以检查边界行和剩余空白。在这一代表性运行中，所选技能将 SpreadsheetBench 留出测试性能从 40.4 提升到 78.9。


## 5 Conclusion
## 5 结论


We presented SKILLOPT, a text-space optimizer that treats an external skill document as the trainable state for frozen LLM agents. By separating the target model that executes tasks from the optimizer that edits skills, and by using bounded edit budgets, minibatch reflection, held-out validation gates, rejected-edit buffers, and epoch-wise slow/meta update, SKILLOPT turns skill improvement into a controlled learning process rather than ad hoc prompt revision. Across six benchmarks, seven target models, and three execution modes, SKILLOPT is best or tied-best on 52 of 52 evaluated cells, lifts GPT-5.5 by +23.5 points on average over no skill in direct chat and by +24.8/ +19.1 points under Codex and Claude Code harnesses, and beats the strongest per-cell baseline from human, LLM, Trace2Skill, TextGrad, GEPA, and EvoSkill skills by +5.4 points on average. Per-benchmark case studies show that these gains arise from compact $( < 2,{000}$ token), interpretable skill artifacts assembled from only 1-4 accepted edits, and that the deployed skills transfer across model scales, harnesses, and nearby benchmarks. These results suggest that compact natural-language skills can serve as a practical domain-adaptation layer for frontier agents, enabling reusable improvement without modifying model weights.
我们提出了 SKILLOPT：一种文本空间优化器，将外部技能文档视为冻结 LLM 代理的可训练状态。通过将执行任务的目标模型与编辑技能的优化器分离，并借助有界编辑预算、小批量反思、保留验证门、拒绝编辑缓冲区，以及按轮次的慢更新/元更新，SKILLOPT 把技能改进转化为受控的学习过程，而非随意的提示修订。在六个基准、七个目标模型和三种执行模式下，SKILLOPT 在 52/52 个评估单元中表现为最佳或并列最佳；相较于直接聊天中“不带技能”，它平均使 GPT-5.5 提升 +23.5 分，并在 Codex 与 Claude Code 的工具链中分别提升 +24.8/+19.1 分；同时，它以平均 +5.4 分的优势击败人类、LLM、Trace2Skill、TextGrad、GEPA 与 EvoSkill 中最强的逐单元基线。按基准的案例研究表明，这些增益来自紧凑的 $( < 2,{000}$ token）、由仅 1-4 次被接受的编辑拼装出的可解释技能工件，并且已部署技能能够在模型规模、工具链以及相邻基准之间迁移。结果表明，紧凑的自然语言技能可以作为前沿代理的可行领域适配层，在不修改模型权重的情况下实现可复用的改进。


Outlook. SKILLOPT optimizes a single skill artifact for a single target domain; natural extensions include skill libraries that share infrastructure across domains, reuse of optimizer-side meta skills across benchmarks, reward-free or preference-driven validation gates for open-ended tasks, and self-distillation of optimized skills back into the target model as a stepping stone toward weight-level adaptation. We hope that treating the skill itself as the trainable object—rather than as a side artifact of prompting-will let future work apply the full toolkit of optimization (learning rates, schedules, regularization, curricula, validation) to a part of the agent stack that has so far been hand-engineered.
展望。SKILLOPT 为单个目标领域优化单一技能工件；自然的扩展包括构建跨领域共享基础设施的技能库、在不同基准间复用优化器侧的元技能、针对开放式任务采用无奖励或基于偏好的验证门，以及将优化后的技能再蒸馏回目标模型，作为迈向权重级适配的垫脚石。我们希望，把“技能本身”视为可训练对象——而非把它当作提示的附属产物——将让未来工作把优化的完整工具箱（学习率、调度、正则化、课程、验证）应用到目前仍主要依赖人工工程的代理栈部分。

## A Additional Method Details and Optimizer Prompts
## A 额外的方法细节与优化器提示


This appendix gives the executable details behind SKILLOPT. The optimization loop keeps the task-execution model fixed and trains only a text skill document. A separate optimizer model reads rollout evidence, proposes patch-style edits, merges and ranks the edits, and submits each candidate skill to a held-out selection gate. The task-execution model only receives the current skill and the benchmark task; it does not see the optimizer prompts below.
本附录提供了 SKILLOPT 的可执行细节。优化循环保持任务执行模型不变，只训练一个文本技能文档。独立的优化器模型会读取回合证据，提出类似补丁的编辑，合并并对编辑进行排序，然后将每个候选技能提交给一个用于选择的隔离门。任务执行模型只接收当前技能和基准任务；它不会看到下面的优化器提示。


## B Limitations
## B 局限性


SKILLOPT studies skill optimization as a lightweight alternative to model-weight adaptation, but it still has several practical limitations. First, the optimization loop relies on scored trajectories and a held-out selection split, so it is most directly applicable when the target task has automatic verifiers, exact-match metrics, executable checks, or otherwise reliable feedback signals. For open-ended domains where success is subjective, multi-dimensional, or costly to judge, the validation gate may require stronger human or model-based evaluation. Second, although the deployed artifact is only a compact best_skill.md, training the skill requires additional rollout computation and calls to an optimizer model; this cost is amortized when the same skill is reused, but may be less attractive for one-off tasks. Third, SKILLOPT intentionally optimizes a single portable skill rather than growing a large skill library or changing model weights. This design improves deployment simplicity, but a single skill may be insufficient for highly heterogeneous domains that require many disjoint procedures. Finally, optimized skills can encode domain-specific heuristics from the training distribution, so careful held-out evaluation remains necessary before transferring them to substantially different models, harnesses, or task settings.
SKILLOPT 将技能优化研究为对模型权重适配的轻量替代方案，但仍存在若干实际局限。首先，优化循环依赖带评分的轨迹和一个保留的选择划分，因此在目标任务具备自动验证器、精确匹配指标、可执行性检查，或其他可靠的反馈信号时，最直接适用。对于成功主观、多维或难以评估的开放式领域，验证门可能需要更强的人类或基于模型的评估。其次，尽管部署产物仅是一份紧凑的 best_skill.md，但训练该技能还需要额外的回合计算，并调用优化器模型；当重复使用同一技能时，这部分成本会被摊销，但对于一次性任务可能吸引力较低。第三，SKILLOPT 有意优化的是单个可移植技能，而不是扩展大型技能库或更改模型权重。该设计提升了部署的简洁性，但对于需要许多彼此独立流程的高度异构领域，单一技能可能不够。最后，优化后的技能可能会从训练分布中编码领域特定的启发式规则，因此在将其迁移到显著不同的模型、工具箱或任务设定之前，仍需要谨慎的保留集评估。


## C Experimental Protocol Details
## C 实验协议详情


Benchmarks and metrics. We use each benchmark's native evaluator and report hard success or exact-match accuracy on held-out test examples. SearchQA measures extractive question answering; SpreadsheetBench evaluates spreadsheet-oriented code and tool use; OfficeQA and DocVQA test local-document and multimodal-document reasoning; SealQA stresses noisy retrieval; LiveMathematicianBench evaluates mathematical multiple-choice reasoning; and ALFWorld tests sequential decision making. Dataset-backed benchmarks use deterministic train/selection/test splits, with a default 2:1:7 split when no benchmark-specific split is stated. The selection split is used only for model selection over candidate skills; all headline scores are computed on held-out test data.
基准与指标。我们使用各基准自带的评测器，并在留出的测试样例上报告硬性成功率或完全匹配准确率。SearchQA衡量抽取式问答；SpreadsheetBench评估面向电子表格的代码与工具使用；OfficeQA和DocVQA测试本地文档与多模态文档推理；SealQA强调噪声检索；LiveMathematicianBench评估数学多项选择推理；ALFWorld测试顺序决策。带数据集支持的基准使用确定性的训练/选择/测试划分；若未说明基准特定划分，则默认采用2:1:7划分。选择集仅用于在候选技能中进行模型选择；所有主结果分数都基于留出的测试数据计算。


Baselines. The no-skill baseline evaluates the frozen student without an optimized skill document. Human-skill and LLM-skill baselines use manually written and one-shot generated skills under the same evaluation protocol. Trace2Skill mines skill artifacts from training trajectories and evaluates the frozen student without SKILLOPT's iterative validation gate. TextGrad and GEPA are reflective prompt-optimization baselines for direct-chat settings. EvoSkill is included for the harness-backed comparison where a matched completed run is available. Entries not measured under the final aligned protocol are marked as -rather than mixed with incompatible runs.
基线。无技能基线在未优化技能文档的情况下评估冻结的学生模型。Human-skill和LLM-skill基线在相同评测协议下使用人工编写和一次生成的技能。Trace2Skill从训练轨迹中挖掘技能产物，并在不使用SKILLOPT迭代验证门控的情况下评估冻结的学生模型。TextGrad和GEPA是用于直接对话设置的反思式提示优化基线。EvoSkill包含在有匹配完成运行可用的harness支持比较中。未在最终对齐协议下测得的条目标记为-，而非与不兼容的运行混在一起。


Optimization protocol. Unless otherwise stated, SKILLOPT runs for four epochs with rollout batch size 40, reflection minibatch size 8, textual learning rate 4, cosine learning-rate decay with minimum rate 2, held-out validation gating, slow update enabled with 20 sampled examples, and optimizer-side meta skill enabled. The optimizer analyzes successes and failures separately, proposes patch-style skill edits, merges duplicate or contradictory proposals, ranks edits under the current learning-rate cap, and applies the selected edits to form a candidate skill. The candidate is evaluated on the selection split and is accepted only if it improves the current selection score; the best accepted skill is exported as best_skill.md. The student model, backend, harness, and benchmark evaluator remain fixed during optimization.
优化协议。除非另有说明，SKILLOPT运行四个epoch，rollout批大小为40，reflection小批大小为8，文本学习率为4，采用最小速率为2的余弦学习率衰减、留出验证门控、启用slow update并使用20个采样样例，以及启用优化器侧meta skill。优化器分别分析成功与失败，提出补丁式技能编辑，合并重复或矛盾的提议，在当前学习率上限下对编辑排序，并应用所选编辑形成候选技能。候选技能在选择集上评估，且仅当其提升当前选择分数时才被接受；最佳已接受技能导出为best_skill.md。优化过程中，学生模型、后端、harness和基准评测器保持不变。


Ablation protocol. One-factor ablations vary a single scalar or component while holding the remaining optimizer configuration fixed. The train-size ablation fixes the train/selection/test split to 2:1:7 and varies how much of the training partition is exposed to the optimizer. The 100% row uses the full training partition under the same split, so it is directly comparable to the smaller-subset rows. Component ablations remove or alter one mechanism at a time, including the edit budget, rejected-edit buffer, and epoch-wise slow/meta update.
消融协议。单因素消融在保持其余优化器配置固定的情况下，改变单个标量或组件。训练规模消融固定训练/选择/测试划分为2:1:7，并改变向优化器开放多少训练部分。100%一行在相同划分下使用完整训练部分，因此可直接与更小子集的各行比较。组件消融一次移除或修改一个机制，包括编辑预算、被拒编辑缓冲区，以及按epoch进行的slow/meta更新。
